# Collect plan-proposal and next-action data for the dashboard

## Objective

Add a read-only data-source layer that reports, for every plan directory
under `plans/`, whether it has been adopted, plus an ordered list of plain
next actions derived from Task lifecycle state and pending plan adoption.
This is data collection only: no HTML, no rendering, and no changes to any
existing dashboard output.

## Acceptance Criteria

- A function reads every immediate subdirectory of `<root>/plans`, parses
  its `PLAN.md` front matter, and reports each plan's id, profile, and
  total proposal count (the count of its `P-##.md` files).
- The function reports whether each plan is still pending adoption (its
  `PLAN.md` body still contains at least one `P-##` placeholder reference)
  or fully adopted (no placeholder references remain in the body), without
  relying on `.aios/tasks` filenames as the signal.
- A missing `plans/` directory yields an empty list rather than an error.
- A plan directory whose `PLAN.md` is missing or fails to parse is reported
  as a named error entry (naming the plan directory and the reason) instead
  of aborting collection of the other plans.
- A second function derives an ordered list of next actions from already
  computed Task rows (a Task in `state: approval` with no decision file
  yet, a Task in `state: blocked`) and from the pending-adoption plans
  above; each next action is a short human-readable string paired with the
  Task or plan id it concerns.
- The next-actions list is an empty array (not omitted or null) when there
  is nothing to act on.
- Automated tests cover: a repository with only adopted plans, only
  pending plans, no `plans/` directory at all, a plan with an unparsable
  `PLAN.md`, and Task-state combinations producing zero, one, and multiple
  next actions.

## Constraints

- Read-only: do not adopt, create, modify, or delete anything under
  `plans/`, `.aios/`, or elsewhere in the repository.
- No server, daemon, watcher, or polling; deliver plain functions invoked
  synchronously by a caller, returning plain data structures.
- Do not render HTML or Markdown output in this proposal.
- Add no new runtime dependencies.

## Context

`src/dashboard.js` already exposes `collectDashboardData`, which loads
every Task, its latest Review, and Worker Session ledger entries for the
existing offline dashboard. `src/plans.js` already parses and validates
`PLAN.md` and proposal files during `adopt`/`inspectPlan` and is the
reference for how plan directories and placeholder references are read.
This proposal adds the missing data source — visibility into plan
proposals that have not yet been turned into real Tasks, plus a
next-actions summary — so a later dashboard page can show current and
upcoming work without re-implementing plan scanning itself.
