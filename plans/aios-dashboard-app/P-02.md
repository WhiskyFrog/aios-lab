# Build the AIOS product dashboard page

## Objective

Extend the existing self-contained, offline dashboard generator into the
full AIOS product surface: a plain-language introduction to AIOS and its
Task/Review/Approval workflow, the current Task lifecycle view, adopted
upcoming Tasks, not-yet-adopted plan proposals, a clear next-actions
summary, and a responsive, accessible layout — consuming the plan-proposal
and next-actions data source delivered earlier in this plan alongside the
existing Task and Worker Session data collection. The result remains one
generated, offline HTML file with no server, daemon, or live polling.

## Acceptance Criteria

- The generated page opens with a short, plain-language introduction
  explaining what AIOS is and how its Task -> Review -> Approval loop
  works, understandable to a reader who has never seen the repository.
- The existing Task lifecycle view (state, retry count/limit, approval,
  latest Review verdict and findings, attempt count) remains present and
  visually distinguishes upcoming/not-yet-done Tasks from done ones.
- A dedicated section lists not-yet-adopted plan proposals, each showing
  its plan id, profile, and proposal count, visually separate from the
  adopted-Task section.
- A dedicated "Next actions" section lists plain-language next actions, or
  shows an explicit empty state when there is nothing to act on.
- A Task, Review, or plan document that fails to load is still rendered as
  a visible, named error rather than aborting the whole page, consistent
  with the existing dashboard's error handling.
- The page stays one self-contained offline HTML file: inline CSS only, no
  required JavaScript for the core view, no network calls, generated by a
  single command with no server, daemon, or watcher.
- Layout is responsive at common viewport widths (for example, a narrow
  mobile width collapses multi-column content to a single column) and uses
  accessible semantics: a correct heading hierarchy, landmark regions,
  sufficient color contrast, and status indicators that do not rely on
  color alone.
- Automated tests cover the new introduction/workflow, upcoming-Tasks,
  plan-proposals, and next-actions sections, plus at least one
  accessibility-relevant structural assertion (for example, a heading
  hierarchy or landmark check).
- Manually opening the generated HTML file in a browser confirms the page
  renders correctly end to end.

## Constraints

- Extend the existing one-shot, self-contained dashboard generation; do not
  add a server, daemon, watcher, or live polling unless this proposal's own
  Attempt demonstrates the goal is unreachable without one.
- Consume the plan-proposal and next-actions data source delivered earlier
  in this plan rather than re-implementing plan-directory scanning.
- Add no new runtime dependency for interactivity; responsive behavior must
  be CSS-only.

## Context

`src/dashboard.js` currently renders an overview, one card per Task, and a
Worker Sessions section as one offline HTML file (documented in the README
Dashboard section); `test/dashboard.test.js` and the README Planner section
describe the existing conventions this proposal must stay consistent with.
The product goal is "build a web dashboard that introduces AIOS and shows
current and upcoming work," and this proposal is the presentation layer
that turns the existing Task data plus the newly collected plan-proposal
and next-actions data into that product.
