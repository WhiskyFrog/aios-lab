# Persist candidate cooldowns and add operator recovery commands

## Objective

Wire the persisted per-candidate cooldown contract into real routed dispatch
— recording a cooldown automatically from corroborated capacity evidence,
reading current cooldowns into every selection, and letting cooldowns expire
on their own — and give the operator `aios route` commands to list active
cooldowns and clear one early when a provider quota resets ahead of schedule.

This Task carries `approval: required` because it changes routed dispatch's
automatic recovery behavior and adds new operator-facing commands that mutate
machine-local runtime state.

## Acceptance Criteria

- A new atomic, machine-local cooldown store persists at (for example)
  `.aios/runtime/candidate-cooldowns.json` using the same discipline as
  `RoutingDecisionLedger`: its schema is validated in full on every load, it
  is written with an atomic replace guarded by an exclusive lock file, and a
  snapshot/compare-and-swap check prevents two writers from silently
  clobbering each other. A missing file is empty state; malformed content is a
  named error and is never silently discarded or repaired.
- When routed dispatch (`src/routing-dispatch.js`) observes a `capacity`
  recovery reason for a candidate — from either the existing structured
  corroboration or the widened textual corroboration this plan adds
  elsewhere — it records or refreshes that candidate's cooldown with the
  corroborated `retry_at` and a bounded, sanitized evidence string produced
  the same way routing-ledger diagnostics are sanitized today. This happens
  alongside, not instead of, the existing capacity-pause routing-decision
  event.
- Every routed selection loads current cooldown state, prunes entries whose
  `retry_at` is at or before the observation time before use — an expired
  cooldown never blocks selection and is dropped from the persisted file on
  the next write it participates in, so the file does not grow unbounded —
  and passes the remaining active records into `selectCandidate`'s cooldown
  input. The considered-candidate audit row recorded in the routing-decision
  ledger names the cooldown gate reason for every skipped candidate.
- `aios route cooldowns [--root <path>]` lists every currently active
  (non-expired) cooldown with its candidate id, `retry_at`, and evidence,
  sorted by candidate id, and exits 0 with an empty list when none are
  active.
- `aios route clear-cooldown <candidate-id> [--root <path>]` removes exactly
  the named candidate's active cooldown (an idempotent no-op, exit 0, when
  none exists), so an operator can unblock a candidate as soon as a quota
  resets ahead of its recorded `retry_at` without editing any JSON by hand.
  The removal is itself an atomic compare-and-swap write like every other
  mutation of this state.
- Both new commands validate the candidate-id shape and `--root` the same way
  existing CLI commands do, follow the same exit-code convention (0 success,
  64 usage error), and never require or accept provider credentials, argv, or
  model identifiers.
- The README is updated to document the new store, its lifecycle (created on
  corroborated capacity, read and pruned at selection time), and the two new
  `aios route` commands, alongside the existing routing-decision ledger
  documentation.
- Tests over fakes and fixture ledgers/cooldown files prove: a corroborated
  capacity event creates a cooldown; a subsequent selection for the same
  action skips the cooled-down candidate with the correct audit reason; an
  expired cooldown no longer blocks selection and is pruned from the stored
  file; `aios route cooldowns` lists exactly the active set; and
  `aios route clear-cooldown` removes one entry and is a no-op when the named
  candidate has no cooldown. No test depends on real elapsed wall-clock time
  or a live provider session.

## Constraints

- Do not let a cooldown record ever contain command argv, credentials,
  environment values, or unsanitized provider stderr; reuse the existing
  bounded, sanitized diagnostic rules.
- Do not let cooldown creation, pruning, or clearing skip the atomic
  lock/compare-and-swap discipline already required for
  `.aios/runtime/routing-decisions.json`.
- Do not let a cleared or expired cooldown rewrite, delete, or otherwise
  affect any routing-decision ledger row; the two stores remain independent
  and this Task does not touch decision-ledger immutability.
- Do not add a daemon, background timer, or any process that prunes cooldowns
  outside of a normal `run`, `progress`, or `route` invocation.

## Context

This Task depends on the cooldown contract, the pure selection gate, and the
widened capacity corroboration defined earlier in this plan, but its own Task
document stands alone; relationships are recorded only in `PLAN.md`'s
Execution Order. It directly answers dogfooding finding 3: after a candidate
exhausts capacity, nothing today stops the router from reselecting it, and
the only existing lever — hand-editing the catalog's `enabled` flag — is
itself the workaround that caused finding 4.
