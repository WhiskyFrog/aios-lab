# Corroborate provider-reported usage-limit failures into capacity events

## Objective

Widen the Codex Worker's capacity corroboration so a session that ends in a
provider-reported usage-limit failure the structured app-server probe cannot
corroborate still surfaces as a `capacity_deferred` outcome carrying the
provider's own reset time, instead of a hard Worker error that consumes the
action's single fallback step. The existing structured app-server probe stays
the preferred corroboration path; a second, fully specified textual parse
engages only when that path is unavailable or inconclusive, and fails closed
into today's hard-failure behavior whenever the provider's own text does not
match a fixed, documented grammar unambiguously.

This Task carries `approval: required` because it changes what evidence a
routed Worker adapter is allowed to convert into a capacity outcome that
influences engine recovery and fallback consumption.

## Acceptance Criteria

- The existing structured app-server corroboration (`queryCodexCapacity` /
  `capacityFromAppServer` in `workers/codex-capacity.mjs`) remains the first
  and preferred corroboration path; its behavior and outcome are unchanged
  whenever it already succeeds.
- When a Codex session ends in a reported failure and the app-server probe
  throws, times out, or returns no corroborated evidence, the Worker attempts
  exactly one additional, deterministic parse of that session's own reported
  error/stream-failure text against a fixed, documented grammar for the Codex
  usage-limit message (the literal usage-limit phrasing plus a trailing bare
  clock-time and/or bare month-day clause), anchored to that session's own
  observed-at time.
- The fallback parse resolves an unambiguous, strictly-future ISO timestamp
  from the recognized clause by choosing the nearest future occurrence of the
  named clock time or month/day relative to the anchor, and only then returns
  a capacity outcome shaped identically to the structured path: a non-empty
  `retry_at` and a `capacity` object with `status: "rejected"`,
  `utilization: 1`, and `resets_at` equal to `retry_at`.
- Any reported text that does not match the fixed grammar exactly — multiple
  candidate dates, a missing reset clause, unrecognized wording, a non-Codex
  error shape, or a clause that resolves to a non-future or ambiguous time —
  is not corroborated. The session keeps today's exact hard-failure message
  and behavior, and no fallback step is consumed by a manufactured capacity
  classification.
- The fallback parse reads only the reported text of the exact session it is
  evaluating; it performs no additional process spawn, network call, or file
  read beyond the existing app-server probe attempt.
- `buildWorkerExecution` continues to accept capacity evidence through its
  existing `capacityEvidence` parameter contract (non-empty `retry_at`,
  `capacity` object) unchanged, so its existing shape checks and downstream
  handling in the engine and session ledger are exercised without
  modification.
- Unit tests over fixed fixtures — including the existing
  `fixtures/codex-usage-limit.ndjson` shape, at least one clock-time-only
  variant, and at least one month-day-only variant — prove: structured
  corroboration wins whenever it is available; the textual fallback correctly
  corroborates a genuine usage-limit message into the right future `retry_at`;
  and a non-matching, ambiguous, or already-past message still fails exactly
  as it does today. No test spawns a real Codex process or opens a real
  app-server session.
- The README section describing Codex Worker capacity handling is updated to
  document the two-tier corroboration order, the fixed textual grammar it
  recognizes, and its fail-closed default.

## Constraints

- Do not weaken, bypass, or change the request/response contract of the
  existing structured app-server probe; it remains the preferred corroboration
  source.
- Do not introduce a general natural-language error classifier. The fallback
  recognizes only the fixed, documented Codex usage-limit grammar and fails
  closed on anything else, including other providers' error text and any
  Codex message shape outside that grammar.
- Do not change `aios.result/v1`, `aios.worker-execution/v1`, or any
  engine-level recovery reason code; this Task only changes which evidence the
  Codex Worker adapter may convert into the existing structured `capacity`
  shape.
- Do not add network calls, credentials, or a live Codex session anywhere in
  test verification.

## Context

Today `workers/codex-worker.mjs` converts only a corroborated
`queryCodexCapacity` app-server response into a `capacity_deferred` outcome;
any other session failure — including the literal usage-limit line Codex
itself prints, captured in `fixtures/codex-usage-limit.ndjson` — falls through
to `reportedError`/`failed` and is treated as a hard Worker error, which
consumes the action's single fallback step in the adaptive router. This was
observed against a live account in dogfooding finding 2 of
`reports/dogfooding-whisky-frog-bundle1.md`.
