# Reconcile distribution-window counting between selection projection and ledger replay after a route reset

## Objective

Reproduce and correct the failure `distribution.observed: must equal the
actual preceding ledger window`, which occurs when a fresh step-0 decision is
recorded for an action whose prior generation was superseded by
`RoutingDecisionLedger.resetAction`. Today, `routing-dispatch.js`'s
`historyProjection` feeds `selectCandidate` a generation-filtered view of the
ledger (via `currentGenerationDecisions`, added in task-0039 attempt 2 to fix
the policy-revision conflict from `review-0055`), so the new record's
`distribution.window`/`observed`/`counts` are computed over that filtered
window. But `validateLedger`'s own per-record distribution replay
(`src/routing-ledger.js` ~line 1140) independently re-derives the same
numbers with a raw positional slice of the full, unfiltered `decisions`
array, which still counts rows belonging to the closed, superseded
generation. The two counts diverge whenever a superseded generation's rows
fall inside the configured window, and the newly recorded row fails
validation the moment it is loaded back.

Make selection-time history projection and the ledger's own
distribution-window validation replay derive `window`/`observed`/`counts`
from one documented, shared generation-counting rule, so a record's
self-reported distribution numbers are always exactly reproducible by
replaying the ledger. This Task carries `approval: required` because it
changes acceptance/validation behavior of the routing decision ledger, a
safety-critical, fail-closed component.

## Acceptance Criteria

- A fixture test reproduces the reported failure end-to-end through the real,
  exported path an operator's `aios run`/`aios progress` uses: a fixture
  ledger with several ordinary preceding routing decisions, a genuine
  `RoutingDecisionLedger.resetAction()` call that supersedes an action's
  current generation, then a fresh selection made by calling
  `historyProjection` and `selectCandidate` exactly as `routing-dispatch.js`
  does and recording the result with `RoutingDecisionLedger.record`. The test
  demonstrates this reproduction fails against the current, uncorrected
  behavior (documented in the Task's Attempt as a red run before the fix, or
  an equivalent before/after demonstration) and passes once the correction
  lands.
- Selection-time distribution-window computation (whatever feeds
  `selectCandidate`'s `history` argument, i.e. `historyProjection`) and
  `validateLedger`'s per-record distribution-window replay
  (`src/routing-ledger.js`) use one named, documented rule for which
  preceding decisions count toward a given record's window — expressed as a
  single shared implementation (a shared helper, or one deriving the other)
  rather than two independently written slicing rules that happen to agree
  only in the no-reset case.
- The shared rule excludes exactly the rows belonging to a partial key's
  *closed* generation (every row superseded, per the existing
  `currentGenerationRows`/`currentGenerationDecisions` definition) as of each
  record's position in append order, and includes every other preceding row —
  including rows from other partial keys, and including a still-open
  generation's own earlier steps — exactly as today's raw slice does in the
  no-reset case.
- No already-recorded decision row, event, or field is deleted, reordered, or
  rewritten to satisfy the corrected rule; `resetAction`'s existing behavior
  (append a `reset` event, flip `status` to `superseded`, otherwise leave the
  row untouched) is unchanged.
- Every existing test in `test/routing-ledger.test.js`, `test/routing-policy
  .test.js`, and `test/routing-dispatch.test.js` continues to pass unmodified,
  demonstrating the corrected rule reproduces today's exact distribution
  numbers for any ledger that has never had a route reset.
- The shared rule is documented with an inline comment at each site that
  applies it (matching the existing comment style around
  `currentGenerationRows`/`currentGenerationDecisions` in
  `src/routing-ledger.js`), naming the invariant so a future change to either
  site is reviewable against the same stated rule.

## Constraints

- Do not weaken or bypass `validateLedger`'s distribution validation (for
  example, by widening the acceptable range or skipping the check for
  post-reset records); the correction must change what is counted, not
  whether counting is enforced.
- Do not change the recorded shape of `distribution` (`window`, `observed`,
  `counts`, `equivalent`, `deficits`, `applied`, `changed_winner`) or any
  other field of `aios.routing-decisions/v1`.
- Do not add network calls, credentials, or a live Codex/Claude session
  anywhere in test verification; use fixture ledgers and a fixture routing
  configuration only.
- Do not touch `.aios/`, existing Tasks, Reviews, or non-fixture
  configuration.

## Context

Dogfooding of the route-reset workflow (`task-0039`, reviewed in
`review-0055` and `review-0056`) found and fixed a policy-revision conflict
in `historyProjection` by filtering through `currentGenerationDecisions`.
That fix leaves `validateLedger`'s independent, generation-unaware
distribution-window replay as the one remaining site that disagrees with the
now generation-aware selection path, producing the failure this Task
reproduces and corrects.
