# Integrate bounded adaptive dispatch with the Role loop

## Objective

Evolve Role resolution so `LoopEngine` can dispatch a Task/Role action through
the deterministic routing policy and recover through a bounded candidate route.
Capacity, timeout, and provider failures prefer an eligible equivalent candidate
from another provider; verification/context failures and repeated evidence can
promote capability; a rejected Review influences the next existing Task attempt.
The integration must preserve Result validation, Task transitions, Review
projection, approval gates, retry limits, provider-specific continuation safety,
and exact legacy Assignment behavior.

This Task carries `approval: required` because it changes the engine's dispatch
and failure-recovery surface and can cause an additional paid Worker session
before returning control to the operator.

## Acceptance Criteria

- Assignment resolution accepts an optional immutable dispatch context containing
  the validated Task, active Role, Task attempt/review generation, applicable
  Review history, run options, and routing policy revision. Existing
  `StaticAssignmentResolver` and `FileAssignmentResolver.resolve(role)` callers
  remain valid; JavaScript's extra argument is ignored in legacy mode and all
  existing direct `LoopEngine.run` tests preserve their outcomes.
- `aios.routing/v1` creates a routed resolver. Before every Role action it
  re-reads and validates configuration, builds workload context, obtains or
  reuses the exact decision-key row, persists the chosen step, and returns a
  normal Worker-compatible dispatch object. Task documents and Worker stdin
  remain unchanged.
- Routed dispatch exposes candidate id/provider/model/tier to the decision
  ledger but still launches the declared argv through the existing
  `CommandWorker`; it does not add a provider SDK or bypass Worker Result and
  session-envelope validation.
- Recoverable signals use structured reason codes, never regex or prose matching:
  `capacity`, `timeout`, `provider_failure`, `verification_failed`,
  `context_insufficient`, and `repeated_evidence`. Existing typed capacity and
  timeout paths map directly. `aios.result/v1` failure payloads may add an
  optional backward-compatible `failure_kind` for `verification_failed` or
  `context_insufficient`; both provider adapters request and validate it. A
  legacy failure without the field remains `worker_reported_failure` and follows
  today's halt behavior rather than being guessed into an escalation.
- Capacity, timeout, and provider failure first choose an unused eligible
  candidate at the same best tier from a different provider. Only when none
  exists may the configured route move upward. Verification failure,
  context-insufficient failure, and the engine's exact duplicate-Attempt-evidence
  detection move directly to a strictly higher tier; they never retry the same
  or a lower tier as a supposed escalation.
- A `changes_requested` Review keeps the existing transition and retry increment.
  On the next Implementer action, workload history raises the minimum tier or
  advances to a stronger unused candidate according to policy. Routing does not
  append a second Review, change `retry.limit`, erase the rejected Review, or
  manufacture an Attempt.
- Reviewer dispatch requires a tier equal to or higher than the Implementer
  decision being reviewed and prefers a different provider. Fallback preserves
  that floor at every step; a Reviewer never falls back below the Implementer
  tier even if it is cheaper or has more capacity.
- Every candidate is attempted at most once per decision action. Configuration
  sets finite positive per-action fallback and per-Task escalation limits, with
  repository defaults that are small and documented. The resolver rejects a
  route containing a cycle. Exhaustion returns the existing engine halt/wait/
  block outcome with a reason naming the bounded routing exhaustion; it never
  loops or silently starts a fresh decision key.
- Provider continuation tokens are candidate-bound. Waiting/resuming the same
  capacity-deferred candidate uses its exact continuation and existing pause/
  duration limits. Cross-provider fallback starts a new session with null
  continuation; no token appears in another candidate's environment or audit
  row. When no cross-provider route remains, existing `--wait-for-capacity`
  semantics decide whether to wait or return `waiting`.
- Each launch, capacity pause, failure classification, fallback/escalation edge,
  completion, and exhausted route atomically updates the matching decision step
  and links to session id when available. The session ledger remains the source
  for usage/cost/capacity telemetry; routing decisions do not double-count it.
- Task state is reread and conflict-checked at the same boundaries as today.
  No failed/fallback dispatch mutates a Task. Only one accepted validated Result
  is projected into an Attempt or Review, and existing Approval handling is
  unchanged.
- Focused tests cover same-tier cross-provider capacity/timeout/provider fallback,
  higher-tier verification/context/repeated-evidence escalation, rejected-Review
  next-attempt promotion, Reviewer tier and provider constraints, candidate-bound
  continuation, fallback exhaustion, cycle rejection, per-action/per-Task limits,
  ledger/session correlation, Task conflict during fallback, and unchanged
  `aios.assignments/v1`, approval, retry, cancellation, and progression behavior.

## Constraints

- Do not introduce concurrent Worker calls. A route is a sequential foreground
  sequence and stops at the first accepted Result or existing terminal gate.
- Do not reset or increase Task retry limits, bypass Review/Approval, parse
  natural-language failure text, reuse a continuation across candidates, or
  rerun a candidate inside one decision action.
- Keep optional structured failure metadata backward compatible with all legacy
  Workers and exact Result validation. Unknown values fail validation; absence
  preserves current behavior.
- Do not add CLI flags, dashboard rendering, a daemon, remote capacity polling,
  a provider SDK, or dependencies in this Task.

## Context

Today `LoopEngine` resolves a Worker with `resolve(role)`, then one
`CommandWorker` owns timeout, capacity continuation, Result validation, and
session recording. Review rejection loops back to Implementer through the
existing retry budget, while exact duplicate evidence now halts before consuming
another retry. The adaptive path must add candidate choice and bounded recovery
around these mechanics, not replace their document and lifecycle authority.
