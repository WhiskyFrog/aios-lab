# Define routing inventory and workload evidence

## Objective

Add a backward-compatible routing configuration contract and a deterministic
provider-neutral workload-context builder. Operators can declare the exact
Claude, Codex, or other command candidates that actually exist, their ordered
capability tiers, Role/capability eligibility, budgets, distribution targets,
and bounded recovery limits without putting any provider or model field into a
Task. Given a Task and active Role, the context builder exposes inspectable
planning/profile/complexity/risk/context/tool/verification/history evidence for
later policy selection and defaults conservatively when evidence is absent.

## Acceptance Criteria

- A strict `aios.routing/v1` parser accepts a versioned JSON configuration with
  exactly these conceptual groups: an ordered tier definition; a candidate
  catalog; policy limits and provider distribution targets; optional
  provider-neutral workload hints selected by Task id or adopted plan id; and
  optional validated operator-override selectors whose application is deferred
  to the operator-control surface.
  Unknown top-level or nested fields are rejected with a path-specific error.
- Every candidate declares a stable candidate id, provider id, exact model id,
  tier, eligible Roles, non-empty command argv, enabled state, context/tool
  capabilities, and cost/latency class. Candidate ids are unique; referenced
  tiers and capabilities must exist; argv uses the existing no-shell validation;
  weights and limits are finite positive values; duplicate provider/model/Role
  entries and an empty eligible catalog are rejected.
- Model identifiers and capabilities come only from the configuration. No
  source default silently invents a current Claude or Codex model, derives a
  tier from a model-name substring, probes a remote model list, or places a
  user-local executable path in repository code.
- The configuration can express at least two provider families and tests use
  explicit Claude and Codex fixtures at lower and higher tiers. Provider ids
  remain data rather than a hard-coded two-value enum so compatible command
  Workers are not excluded.
- A workload-context builder receives the complete validated Task, active Role,
  repository root, routing config, and optional Review/session history and
  returns a normalized immutable object containing: Task id; Role; work kind
  (`planning`, `implementation`, or `unknown`); parent plan id/profile when
  unambiguously discoverable; complexity and risk bands; Task-context size;
  required capabilities/tools; verification burden; latency and cost budget;
  approval/retry state; relevant prior outcome counters; uncertainty flags; and
  a source label for every non-default value.
- Parent-plan discovery reuses validated adopted-plan inspection and succeeds
  only when exactly one adopted plan names the Task. Zero or multiple matches
  are recorded as uncertainty rather than guessed. A malformed plan fails that
  assessment closed without making a lower tier eligible.
- `work_kind: planning` is established only by an explicit provider-neutral
  hint or a strict plan-only Task contract (an Objective naming its plan output,
  a constraint limiting writes to that plan directory, and a required
  non-mutating `adopt --check` verification). A title containing "plan" alone
  is insufficient. Conflicting evidence becomes `unknown` and is marked
  conservative.
- Complexity, risk, context, and verification rules are deterministic and
  documented. They may use explicit hints, plan profile, Task byte/section
  counts, approval requirement, retry/Review history, declared capabilities,
  and named verification commands; they do not call an AI model or apply an
  opaque score. Missing semantic hints raise uncertainty/risk and can never be
  used as evidence for a lower tier.
- A planning workload and any workload whose safety-critical evidence is
  unknown carries a normalized `minimum_tier` of the configured high tier. A
  lower-tier minimum is emitted only for a non-planning, bounded context with
  low complexity, low risk, known tool requirements, objective verification,
  and no unresolved failure history. The exact gate and rejection reasons are
  present in the normalized context.
- Configuration loading preserves `aios.assignments/v1` byte-for-byte behavior:
  the existing Role-only file still produces the same `FileAssignmentResolver`,
  is re-read before each Role action, records sessions as before, and emits no
  fabricated routing-decision evidence. Routing activates only when the input
  schema is `aios.routing/v1`.
- Unit tests cover every validation rule, legacy Assignment compatibility,
  planning detection and title-only rejection, adopted/ambiguous/malformed plan
  context, high-tier conservative defaults, lower-tier gate pass/fail cases,
  prior failure/review evidence, and stable normalized output for identical
  repository state.

## Constraints

- Do not change Task, Review, Plan, Result, or Worker-execution document
  schemas. Workload hints live in routing configuration and remain provider and
  model neutral.
- Do not launch a Worker, choose a candidate, write a decision ledger, change
  the Loop Engine, add CLI flags, or render dashboard UI in this Task.
- Reuse existing Assignment argv validation, plan inspection, Task loading, and
  session validation rather than creating alternate document parsers.
- Do not add a provider SDK, network discovery, credential handling, daemon,
  concurrency, or dependencies.

## Context

`src/workers.js` currently validates `aios.assignments/v1` and
`FileAssignmentResolver.resolve(role)` returns one `CommandWorker` after
re-reading the file. `src/plans.js` already validates adopted plans and their
profiles, while Tasks and Reviews already expose approval, retry, and prior
outcome evidence. This Task establishes the explicit inventory and normalized
inputs needed to evolve resolution safely without changing those durable
documents.
