# Correct the inter-Task progression plan after operator review

## Objective

Revise the existing `software-feature` plan under
`plans/inter-task-progression/` so its engine-contract approval policy and
dashboard observability claims are internally consistent, mechanically valid,
and ready for a new independent Review after the operator declined to approve
the first version.

## Acceptance Criteria

- Preserve the plan's narrow goal: an operator-invoked, deterministic,
  resumable sequential runner over the real Task order in an adopted
  `PLAN.md`, reusing the existing single-Task Loop Engine and stopping at every
  human, Worker, capacity, cancellation, invalid-document, blocked, and
  repository-conflict boundary.
- Keep the minimal additive machine-readable Loop Engine outcome category
  proposed by the first plan, because callers must not classify free-text
  reasons. Specify its stable categories and backward-compatibility boundary
  precisely enough for implementation and regression testing.
- Any proposal that changes `LoopEngine` behavior or its returned outcome
  contract carries `approval: required`, not only the later CLI proposal, in
  accordance with the repository policy for engine-surface changes.
- Reconcile the read-only dashboard proposal with the decision not to add a
  persisted plan-run record. The dashboard may report current state derived
  from plan and Task documents and clearly labelled last-observed evidence
  from the existing session ledger, but must not claim it can reconstruct
  transient cancellation or repository-conflict events after the process has
  ended. It must distinguish directly observed current state from historical
  session evidence.
- The dashboard still reports plan id, ordered Tasks, completed count, current
  Task, approval/blocked/invalid-document state, and exact operator action when
  those values are derivable read-only. Cancellation remains CLI-only.
- The CLI proposal provides an explicit, non-conflicting exit-code mapping for
  plan complete, awaiting approval, blocked, capacity wait, cancellation,
  other halt, and usage error rather than deferring that policy to its
  Implementer.
- Preserve focused decomposition and ordering. The existing four-proposal
  shape may remain if the correction keeps each Task independently
  implementable and reviewable; any structural change must be justified in
  `PLAN.md`.
- Adaptive Claude/Codex model routing remains excluded from this plan and
  preserved as the separate follow-on roadmap goal.
- `node src/cli.js adopt plans/inter-task-progression --check` passes without
  writing Tasks, and the Implementer's Verification records the exact result.
- The new Reviewer explicitly checks the three operator corrections above:
  approval on every engine-contract-changing proposal, honest dashboard
  observability without hidden run state, and a complete CLI exit-code map.

## Constraints

- This is a corrective planning Task. The Implementer writes only under
  `plans/inter-task-progression/`; it does not implement progression or modify
  `.aios/`, `src/`, `test/`, workers, assignments, or git history.
- Do not add a persisted plan-run record, daemon, watcher, concurrent
  scheduler, Task dependency field, or mutating dashboard control.
- Do not erase the first plan's useful reasoning merely to make a smaller
  diff; preserve valid decisions and correct only what the operator review
  identified or what must change for internal consistency.
- This Task carries `approval: required` because its output authorizes a plan
  containing engine execution and contract changes.

## Context

`task-0017` produced `PLAN.md` and P-01 through P-04, and `review-0019` passed
their original acceptance criteria. The operator withheld approval after an
independent audit found that P-01 changes the Loop Engine outcome contract but
was marked `approval: not_required`, and that P-03 promised dashboard parity
for transient stop events despite the plan deliberately persisting no run
record. The audit supports the plan's additive machine-readable outcome code
and its no-new-run-state decision; this Task makes the approval boundary and
observability claims honest, and removes ambiguity from CLI exit codes.
