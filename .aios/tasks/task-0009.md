# Resume capacity-paused Worker sessions automatically

## Objective

Let one foreground Loop Engine run observe each Claude session's usage and
authoritative quota-reset time, pause safely when capacity is exhausted, and
resume the same session after refill without another operator trigger.

## Acceptance Criteria

- `aios run <task-id> --wait-for-capacity` keeps one foreground run alive when
  a Worker reports a structured capacity deferral. It waits outside the child
  Worker's execution timeout and re-invokes the same resolved Worker with its
  opaque continuation after the reported reset time.
- The wait is bounded by positive `--max-capacity-wait-ms` and
  `--max-capacity-pauses` options with documented defaults. Cancellation and
  exhausted bounds halt promptly with a clear reason.
- Without `--wait-for-capacity`, a valid capacity deferral returns a distinct
  `waiting` outcome, includes its UTC `retry_at`, and exits with code 75. The
  existing `done`, `blocked`, halted, and usage-error output and exit behavior
  remains compatible.
- Capacity is a provider-neutral control path outside Task, Review, and Result
  v1. The Loop Engine never parses provider prose and never derives refill
  time from token counts. A deferral is eligible for automatic waiting only
  when it carries a finite future reset timestamp and an opaque continuation.
- `workers/claude-worker.mjs` consumes Claude Code `stream-json`, preserves its
  existing Role Result translation, and maps only a structured
  `rate_limit_event` with `status: rejected`, numeric `resetsAt`, and an exact
  `session_id` to capacity deferral. On resume it uses that exact id with
  `--resume`; text-only limit messages and missing or invalid fields remain
  ordinary failures.
- Before waiting and again before resuming, the engine verifies that the
  active Task bytes, state, Role, and Attempt are unchanged. Capacity pauses
  never change Task state, append an Attempt or Review, or increment
  `retry.count`.
- Every structured Claude execution updates an atomically written,
  git-ignored `.aios/runtime/sessions.json` ledger. One row per provider
  session records Task, Role, model, timestamps, invocation count, latest
  outcome, numeric token usage and cost when supplied, and latest capacity
  utilization/reset when supplied. The ledger is operational telemetry, not
  lifecycle truth.
- `aios dashboard` remains one-shot and read-only and adds a clearly separate
  Worker Sessions section showing the ledger's per-session usage, status, and
  refill time. A missing ledger is an empty state; an invalid ledger is a
  visible dashboard error rather than a crash.
- Tests use fake Workers, captured NDJSON fixtures, and injected clocks/sleepers
  to cover completion, deferral then exact-session continuation, multiple
  pauses, bound/cancel behavior, Task mutation during a wait, malformed or
  past reset values, generic failures that never wait, ledger persistence, and
  dashboard rendering. No test makes a network request, invokes a billed
  Claude session, or performs a real quota-length sleep; the full suite passes.

## Constraints

- Do not change the `aios.task/v1`, `aios.review/v1`, or `aios.result/v1`
  contracts or their lifecycle invariants.
- Do not auto-retry authentication, billing, permission, network, timeout,
  malformed-output, context-window, or Result failures. An Implementer may
  already have changed repository files, so ambiguous failures still require
  operator inspection.
- Keep the existing `execute(Task) -> Result` Worker boundary from the Loop
  Engine's point of view; continuation and telemetry belong to the command
  transport layer.
- One Task and one foreground supervisor only. No daemon, watcher, queue,
  Redis, event bus, multi-task scheduler, lease, OS wakeup, or reboot-safe
  automatic continuation.
- No hard-coded quota-window calculation and no new runtime dependency.
- Do not commit or push from a Worker session.

## Context

The current Loop Engine already advances one Task through Roles in a single
foreground command, but any Worker error halts it. The Claude adapter starts a
fresh JSON session per Role and currently discards the outer session id,
usage, cost, and rate-limit information. Claude Code 2.1.205 supports
`stream-json` output and exact-session `--resume`. Its documented structured
`rate_limit_event` supplies `status`, optional `resetsAt`/`utilization`, and
`session_id`; its final Result supplies the session usage envelope.

Task documents remain the source of lifecycle truth. The runtime session
ledger may explain why execution is waiting and when it can continue, but it
must never move the Task or substitute for Review evidence.
