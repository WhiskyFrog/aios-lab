# Prevent same-policy exact-key collisions between a superseded generation and a fresh post-reset record

## Objective

Reproduce and correct (or explicitly, fail-closedly prevent) the case where
`RoutingDecisionLedger.record()` rejects a legitimate fresh step-0 selection
recorded after a real `resetAction()` when the routing policy revision is
unchanged across the reset. `record()` locates a candidate row's "existing"
slot with `findRecordIndex(snapshot.decisions, record.key, record.step)`,
which matches on `(task, role, attempt, policy_revision, step)` alone, with
no notion of generation or row status. `resetAction` never requires or
forces a policy-revision change — it only supersedes a generation's rows and
appends a `reset` event — so an operator who resets a stuck action without a
routing-policy edit produces a fresh step-0 record whose key is identical in
every field to the superseded generation's own step-0 row. `findRecordIndex`
finds that superseded row as the "existing" one; since its content differs
from the fresh record, `record()` throws "Refusing to rewrite recorded
decision ... step 0" instead of accepting the fresh generation, even though
`resolveKey` and `record()`'s own open-sequence check already treat that
generation as closed.

Make `record()`'s exact-match lookup generation-aware, so a row belonging to
a closed generation is never mistaken for the "existing" row a fresh
generation's record must be compared against — regardless of whether the
policy revision changed across the reset — or, if some variant of this case
cannot be supported safely, make `resetAction`/`record`/the CLI reject it
explicitly and fail closed rather than silently letting the CLI report reset
success for a sequence `record()` will later refuse. This Task carries
`approval: required` because it changes acceptance behavior of the routing
decision ledger's immutability guard, a safety-critical, fail-closed
component.

## Acceptance Criteria

- A fixture test reproduces the collision end-to-end through the real,
  exported path: a fixture ledger with an action's generation taken through
  `selected` → `dispatched` → `failed`, a genuine
  `RoutingDecisionLedger.resetAction()` call that supersedes it *without*
  changing the routing configuration (so the fresh selection's
  `policy_revision` is identical to the superseded generation's), then a
  fresh selection made via `historyProjection` + `selectCandidate` and
  recorded via `RoutingDecisionLedger.record`. The test demonstrates this
  reproduction fails against current behavior and passes once corrected.
- `record()`'s exact-match lookup never treats a row belonging to a closed
  generation (per the existing `currentGenerationRows` definition already
  used by `resolveKey` and `record()`'s own open-sequence check) as the
  "existing" row for a fresh generation's record at the same
  `(task, role, attempt, policy_revision, step)` — whether or not the policy
  revision changed across the reset.
- Existing, still-open-generation behavior is unchanged: recording an
  identical record twice within the same open generation remains an
  idempotent no-op, and any attempt to actually rewrite a still-open
  generation's already-recorded row with different content is still rejected
  with a clear error.
- The CLI's `aios route reset` success path and its README documentation
  accurately reflect the corrected contract: either a subsequent real
  `aios run`/`aios progress` selection can be recorded successfully
  regardless of whether the policy revision changed across the reset, or any
  scenario the correction does not extend to is named explicitly in the CLI
  output and README rather than left to surface later as an unexplained
  `record()` rejection.
- Every existing test in `test/routing-ledger.test.js` and
  `test/routing-reset.test.js` continues to pass unmodified, including its
  existing changed-policy-revision reset-then-reselect coverage, demonstrating
  the correction is additive to the same-policy case and not a behavior
  change to the already-covered changed-policy case.
- New regression coverage exercises the same-policy reset-then-reselect flow
  through the real dispatch/record path (not an isolated
  `RoutingDecisionLedger` API call standing in for it), matching the style of
  `test/routing-reset.test.js`'s existing changed-policy coverage.

## Constraints

- Do not weaken `record()`'s protection against rewriting an already-recorded
  row of a still-open generation; the correction narrows what counts as
  "existing," it does not loosen what happens once a true collision within an
  open generation is found.
- Do not change `resetAction`'s existing behavior of appending only a
  `reset` event and flipping `status` to `superseded`; do not make
  `resetAction` require or infer a policy-revision change.
- Do not add a generic ledger-editing or history-rewriting command; any CLI
  message change is limited to accurately describing the reset/record
  contract's actual scope.
- Do not add network calls, credentials, or a live Codex/Claude session
  anywhere in test verification; use fixture ledgers and a fixture routing
  configuration only.
- Do not touch `.aios/`, existing Tasks, Reviews, or non-fixture
  configuration.

## Context

`test/routing-reset.test.js` (from task-0039) only exercises a reset
followed by a fresh selection under a *changed* policy revision
(`policy-v2`); the same-policy path through `record()`'s exact-match lookup
has never been exercised, even though `resetAction` itself never requires a
policy change. This is the gap the Brief's "also cover or explicitly prevent
same-policy exact-key collisions" clause names.
