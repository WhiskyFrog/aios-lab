# Define the Task document format

## Objective

Establish the canonical Task document contract that Sprint 0 will use as the
source of truth for the Implementer-to-Reviewer loop.

## Acceptance Criteria

- The Task has a stable identity and Project association.
- Machine-owned lifecycle state is separate from the human-readable brief.
- The state model covers implementation, review, bounded retries, optional
  human approval, successful completion, and retry exhaustion.
- A Task selects a permanent Role without naming its current worker.
- Review evidence is referenced without embedding Review history in the Task.
- Worker Results remain available in an append-only Attempts section.
- A reusable template and precise validation rules are present.

## Constraints

- Define only the Task document contract.
- Do not implement the Loop Engine.
- Do not define Project, Role, Assignment, Review, or worker adapter formats.
- Do not add fields without a requirement in the current loop.

## Context

The repository began empty. The project vision supplied for Sprint 0 is the
sole design authority: the organization owns the workflow, Roles are
permanent, workers are replaceable, and Task files are the source of truth.

## Attempts

### Attempt 1

#### Summary

Defined the v1 Task contract, its lifecycle and invariants, and a reusable Task
template under `.aios/tasks/`.

#### Verification

Checked the contract against the Sprint 0 Implementer, Reviewer, retry, and
optional human approval flow. Confirmed that no worker identity or execution
mechanism is stored in the Task.
