# Add an operator route-reset command for stranded actions

## Objective

Give the operator an explicit, auditable `aios route reset
<task-id>:<role>:<attempt>` command that supersedes every recorded
routing-decision row for exactly one action key when a routing-policy change
has stranded it, so the next selection for that key can proceed under the
current policy revision without hand-editing
`.aios/runtime/routing-decisions.json`, while every existing row remains
untouched history.

This Task carries `approval: required` because it adds a new operator command
that mutates the routing-decision ledger's outcome state directly, outside the
normal dispatch/recovery path.

## Acceptance Criteria

- `aios route reset <task-id>:<role>:<attempt> [--root <path>]
  [--reason <text>]` locates every recorded row for that exact partial key
  (task, role, attempt) across however many steps it has, and refuses with
  exit 1 and no write unless every one of those rows currently has status
  `selected`, `dispatched`, or `failed` — never when any row is already
  `completed` or already `superseded` — matching the Brief's "all non-terminal
  or failed" scope exactly.
- A successful reset appends a new terminal outcome/event to every one of
  those rows moving its status to `superseded`, recording the supplied or a
  default reason, an explicit policy-changed audit trail, and an
  `observed_at` that is not before that row's current `observed_at`. No
  existing field on any row is rewritten in place, preserving the ledger's
  immutable-history invariant.
- After a reset, resolving or recording that exact partial key under the
  current policy revision no longer raises the policy-revision conflict that
  caused dogfooding finding 4; a fresh selection for that key can be recorded
  as step 0 of a new, unrelated decision sequence exactly as if the action had
  never run before, without deleting the superseded rows.
- The ledger's `updated_at` invariant (equal to the latest recorded
  `observed_at` across all decisions) and every other existing load-time
  validation in `src/routing-ledger.js` continue to hold after a reset; the
  post-reset file passes every existing ledger and snapshot validation with no
  schema relaxation.
- The command performs its own load-modify-atomic-write compare-and-swap cycle
  against `.aios/runtime/routing-decisions.json`, reusing the ledger's
  existing lock/snapshot discipline, fails closed on a concurrent conflicting
  write, and never touches Task documents, Reviews, approvals, or the session
  ledger.
- The command is rejected with exit 64 for a key that does not exist, a
  malformed `<task-id>:<role>:<attempt>` selector, or a key whose rows are not
  all non-terminal or failed, each with a distinct, actionable error message;
  nothing is written on any rejection.
- The README documents the command, exactly which rows it can and cannot
  reset, and that it is the intended remedy for a routing-policy revision
  change stranding an in-flight action — replacing the hand-editing
  workaround recorded in dogfooding finding 4 — while noting that an
  already-recorded, still-immutable decision (dogfooding finding 5) is
  addressed by resetting the action, never by mutating or reattaching an
  override to the old row.
- Tests over fixture ledgers prove: reset succeeds and unblocks a fresh
  same-key selection when every row is selected, dispatched, or failed; reset
  is refused when any row is already completed; reset is refused for an
  unknown key; the post-reset ledger still passes every existing ledger
  validation; and a concurrent lock conflict fails closed with no write.

## Constraints

- Do not delete, rewrite in place, or renumber any existing decision row; a
  reset only appends a terminal status/event, never removes history.
- Do not allow reset to succeed when any step for the key is already
  `completed`; a finished action can never be silently redone.
- Do not let this command attach, change, or reinterpret an override on an
  existing row — that remains correctly immutable — and do not let it accept
  a new override as part of the reset; it only clears the way for a brand-new
  selection.
- Do not add a generic ledger-editing or ledger-deletion command; the surface
  stays narrow to this one recovery operation.

## Context

`RoutingDecisionLedger.record` rejects a new row for a partial key whose
existing rows carry a different policy revision, and resolving that key
raises the same conflict on read. Dogfooding finding 4 hit this after an
ordinary routing-config edit and could recover only by hand-editing the
ledger file twice, including manually restoring the `updated_at` invariant.
This Task is the plan's only Task the operator invokes by hand between loop
runs rather than something the engine exercises automatically, and it is
independent of the other Tasks in this plan.

### Attempt 1

#### Summary

Implemented `aios route reset <task-id>:<role>:<attempt> [--root <path>] [--reason <text>]`. In src/routing-ledger.js: added `RoutingDecisionLedger.resetAction()` (atomic load-modify-CAS write reusing the existing lock/#commit discipline) which supersedes every row of a partial key's current decision sequence, appending a `reset` event per row with a fixed `policy_changed` reason code plus the supplied/default diagnostic, and refuses closed (no write) for an unknown key or any row not in selected/dispatched/failed. Introduced a `currentGenerationRows` helper (splits a partial key's rows at its last step-0 row) and rewired `validateLedger`'s step/policy-revision checks, `record()`, and `resolveKey()` to be generation-aware, so once every row of a key's sequence is superseded, `resolveKey` returns null and `record` accepts a fresh step-0 under the current policy revision — fixing dogfooding finding 4 — while every existing row/field stays untouched (finding 5's immutability is preserved; no override is ever attached/reinterpreted). Added `policy_changed` to FAILURE_REASON_CODES and `reset` to ROUTING_EVENT_KINDS (additive, no schema relaxation). Added the CLI subcommand in src/cli.js (parsing, --help/usage text, handler mapping RoutingLedgerError to exit 64). Documented the command in README.md under a new 'Resetting a policy-stranded action' section. Added test/routing-reset.test.js (9 tests) covering: reset unblocks a fresh same-key selection under a new revision; refusal on a completed row; refusal for an unknown key; fail-closed on a concurrent lock conflict with no write; override immutability; CLI --help text; CLI success path; CLI exit-64 paths (malformed selector, unknown key, non-resettable) writing nothing; CLI usage errors. Note: the Task's AC1 says exit 1 for the 'not all rows resettable' case while AC6 says exit 64 for the same case (plus unknown key/malformed selector) — I followed AC6 and the pre-existing 'route: 0 success, 64 usage error' convention already documented for `route cooldowns`/`route clear-cooldown`, returning 64 uniformly for all three rejection categories. Also flagging a discovered, out-of-scope gap: `routing-policy.js`'s `validateHistory` (fed via `historyProjection` in routing-dispatch.js) independently rejects mixed-revision history for one partial key, so a real `aios run`/`aios progress` right after a reset can still hit a related conflict there; fixing it would require changing routing-policy.js's tested forgery-detection contract, which is outside this Task's stated 'independent of other Tasks' / narrow-surface scope — the Task's own ACs and Context scope this work to RoutingDecisionLedger + CLI + fixture-ledger tests, which is what's implemented and verified.

#### Verification

Ran the full suite (`npm test`): 394/394 passing, twice in a row (one earlier run showed 1 failure in the pre-existing, previously-documented flaky test `CommandWorker waits for a timed-out descendant tree to terminate`, unrelated to this change — it passed cleanly on rerun). Also manually exercised the CLI end-to-end via a scratch fixture ledger: seeded a selected→dispatched→failed action, confirmed `resolveKey` under a new policy revision threw the pre-fix conflict, ran `aios route reset task-x:implementer:1 --reason ...` (exit 0), confirmed the row became `superseded` with a `reset` event, confirmed `resolveKey` under the new revision now returns null, confirmed a fresh step-0 `record()` under the new revision succeeds and the reloaded ledger still passes full validation with the correct `updated_at`, and confirmed exit 64 for unknown key / malformed selector / a completed row present, in each case verifying the underlying file was unchanged.

### Attempt 2

#### Summary

Fixed the review-0055 blocking finding: routing-dispatch.js's real dispatch/recovery paths still hit the policy-revision conflict after a route reset, because historyProjection fed selectCandidate the entire unfiltered snapshot.decisions, including a reset key's now-superseded generation. Added an exported `currentGenerationDecisions(decisions)` helper in src/routing-ledger.js (reusing the existing currentGenerationRows split) that drops a partial key's rows entirely once its current generation is fully superseded — mirroring resolveKey's 'resolves as null' semantics — and wired it into routing-dispatch.js's `historyProjection` (now exported for testing), which both selectCandidate call sites (the initial resolve path and the #advance recovery path) already funnel through. Added a regression test in test/routing-reset.test.js that reproduces the real bug via the actual exported historyProjection with a genuine superseded row from a real resetAction() call (not a decoy), asserting a fresh same-key selection at a new policy revision no longer throws.

Also confirmed no other source files were modified besides src/routing-dispatch.js and src/routing-ledger.js (plus the new test file).
