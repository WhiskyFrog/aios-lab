# Prove inter-Task progression end to end and document it

## Objective

Prove the assembled system — the core progression library, the `aios
progress` CLI subcommand, and the dashboard's Plan Progress section, all
delivered earlier in this plan — actually works together end to end, and
document the new command in the README so an operator can discover and use
it the same way they already discover `run`, `dashboard`, and `adopt`.

## Acceptance Criteria

- A disposable end-to-end demonstration proves two adopted Tasks complete in
  order from one operator command: using a temporary scratch plan directory
  and temporary `.aios/` state (not committed to the repository, or fully
  torn down afterward), adopt a two-Task plan with real `aios adopt`, assign
  Workers that deterministically succeed, run `aios progress` exactly once,
  and observe both Tasks reach `state: done` in their intended order without
  a second operator invocation. The Implementer's Verification records
  exactly what was run and what was observed; the demonstration's temporary
  artifacts are not left behind in the repository.
- At least one automated integration test (durable, committed, distinct from
  the disposable demonstration above) exercises the real `aios adopt` then
  `aios progress` command sequence against a temporary fixture repository
  and asserts the reported stop reason and operator action for a
  representative non-success case (for example, an approval gate) in
  addition to the full-success case, proving the CLI, the core library, and
  Task/Review documents interoperate correctly end to end rather than only
  through mocked units.
- The dashboard's Plan Progress section (delivered earlier in this plan) is
  exercised against the same or an equivalent fixture used for the
  non-success integration test above and confirmed to agree with `aios
  progress`: for a current-state stop reason (awaiting approval, blocked, or
  invalid document) the dashboard's reported category and operator action
  match `aios progress`'s reported stop reason and operator action exactly;
  for a stop reason outside those three current-state categories, the
  fixture confirms the dashboard shows only clearly labelled last-observed
  session-ledger evidence (or nothing, if no ledger row exists yet) rather
  than asserting it reproduces `aios progress`'s live stop reason or
  operator action text; and the fixture confirms the dashboard never renders
  cancellation or repository-conflict information. This proves the two
  surfaces agree wherever both can honestly speak, and that the dashboard
  never overclaims where they can't.
- The README gains a section documenting `aios progress`: usage, its
  options and their defaults, its exit codes, its cancellation behavior, and
  its idempotent-resume guarantee, written in the same style as the existing
  Run/Capacity Wait/Dashboard/Planner sections.
- The README's Dashboard section is updated to mention the new Plan Progress
  content (or this criterion is satisfied by the dashboard proposal earlier
  in this plan if it already did so; this proposal does not duplicate that
  update).
- The README's Roadmap section is updated to reflect that inter-Task
  progression is now implemented, preserving the existing description of
  the adaptive-model-routing follow-on milestone as still separate and
  future work.
- Every automated test added across this plan (core library, CLI, dashboard,
  and this proposal's integration tests) passes together in one full test
  suite run, confirming no cross-proposal regression.

## Constraints

- This proposal does not implement new progression behavior; it only
  demonstrates, integration-tests, and documents behavior delivered by the
  earlier proposals in this plan. Any gap it finds must be reported, not
  silently patched around, if the fix would belong to an earlier proposal's
  scope.
- The disposable demonstration must not leave temporary Tasks, Reviews,
  approvals, or plan directories committed in the repository.
- Do not weaken, skip, or work around any Review or Approval gate to make
  the demonstration succeed faster.

## Context

This is the last proposal in the plan; it depends only on execution order,
not a dependency field, on the core library, CLI subcommand, and dashboard
section delivered earlier. The retrospective context that motivated
task-0017 — a Task's Implementer timing out and needing manual completion —
means this proposal's demonstration should also implicitly show that a stop
condition (not just full success) is reported clearly enough for an operator
to recover without guessing, which is why a non-success case is required
alongside the two-Task success demonstration.
