# Connect Claude Code sessions as command Workers

## Objective

Provide a worker adapter that lets the Loop Engine dispatch the Implementer
and Reviewer Roles to non-interactive Claude Code sessions through the
existing command Worker contract, without any engine change.

## Acceptance Criteria

- A worker adapter under `workers/` receives the Task document on stdin,
  derives the Role from `AIOS_ROLE`, launches one non-interactive Claude
  Code session, and writes exactly one valid `aios.result/v1` JSON object to
  stdout.
- Role prompts instruct the session to satisfy the Task within its
  Constraints, forbid modifying `.aios/` documents and git history, and
  require a machine-readable payload reply.
- The Implementer session may edit the repository; the Reviewer session is
  restricted to read-only inspection.
- Unusable session output (crash, timeout, or a reply that yields no valid
  payload) surfaces as a Worker failure that halts the run without a Task
  transition, and a session that reports it cannot complete becomes a
  `status: failure` Result.
- An Assignment file maps `implementer` and `reviewer` to the adapter so
  `aios run` drives a Task end to end with no engine or contract change.
- An end-to-end run in a disposable sandbox repository reaches `done`
  through Claude-backed Implementer and Reviewer sessions.
- README documents the adapter, its permission model, and the trust and
  cost caveats.

## Constraints

- Do not modify `src/` engine behavior or the v1 document contracts.
- Claude-specific behavior stays in `workers/` and Assignment configuration.
- The adapter is trusted local configuration; no secrets in the repository.
- Do not add scheduling, queues, or multi-process execution.

## Context

task-0003 delivered the engine and a generic command Worker: an argv
command receives the Task document on stdin plus `AIOS_TASK_ID` and
`AIOS_ROLE` in the environment, and returns one `aios.result/v1` JSON
object on stdout. The Claude Code CLI provides non-interactive sessions via
`--print` with `--output-format json`. On this machine the native binary
lives behind a PowerShell shim, so the Assignment must name the executable
explicitly. This is the first Assignment that binds a real coding agent to
the permanent Roles.

### Attempt 1

#### Summary

Added workers/claude-worker.mjs, a command Worker adapter that runs one non-interactive Claude Code session per Role action: it reads the Task document from stdin, builds a Role-specific prompt that forbids touching .aios/ and git history, launches the named Claude executable with --print and --output-format json (acceptEdits plus Bash for the Implementer; default read-only permissions plus npm test for Reviewer and Approver), validates the JSON-only reply, and emits one aios.result/v1 envelope, mapping a failure_reason reply to status: failure and any unusable output to a nonzero exit. Added .aios/assignments.json binding implementer and reviewer to the adapter through the native claude.exe, since the PATH shim is PowerShell-only and Workers spawn without a shell. Documented the adapter, permission model, and trust and cost caveats in the README.

#### Verification

End-to-end sandbox run: a disposable repository containing only task-9100 reached done from one aios run invocation - the Claude Implementer session created greet.js, its tests, and package.json, the engine projected the framed Attempt, and the Claude Reviewer session independently ran npm test (3 of 3 passing) and returned a pass verdict that the engine persisted as review-0001. Re-running npm test in the sandbox afterward reports 0 failures, and the engine suite still passes 24 of 24.
