# Implement aios init repository bootstrap

## Objective

Add an `aios init` subcommand that prepares a target repository for AIOS
operation in one idempotent, fail-closed pass: it scaffolds the `.aios/`
layout the engine, dashboard, and progression already expect and installs
machine-local worker assignment (or routing) configuration from an
operator-confirmed source, without ever overwriting existing operator state
and without leaking user-local paths into anything the target repository
would commit.

## Acceptance Criteria

- `aios init [--root <path>] [--from <config-file>] [--check]` joins the
  existing CLI with usage text, `--root` defaulting to the current working
  directory, and exit codes 0 (initialized, already initialized, or
  `--check` passed), 1 (target validation failure), and 64 (usage error),
  consistent with the adopt command's convention.
- The target is validated with the shared target-inspection contracts
  before any write: a path that is not a directory, not a repository root,
  or in a conflicting `.aios/` state produces the corresponding distinct
  error and leaves the target byte-for-byte untouched. All validation,
  including of the `--from` source, completes before the first filesystem
  mutation.
- On success the command ensures this scaffold under the target root:
  `.aios/tasks/`, `.aios/reviews/`, `.aios/results/`, `.aios/approvals/`,
  and `.aios/runtime/` directories; a `.aios/.gitignore` covering
  `runtime/` and the assignment/routing configuration file so machine-local
  state stays out of the target's version control; and, when `--from` is
  given, the validated configuration copied to the engine's default
  location under `.aios/`. The command never edits any file outside
  `.aios/`.
- `--from` accepts an existing configuration file that validates as
  `aios.assignments/v1` or `aios.routing/v1` using the existing loaders.
  Adapter references must be explicit and portable: every command argv
  token containing a path separator must either be an absolute path that
  exists or resolve to an existing file relative to the target root;
  otherwise the command fails before writing, naming the offending Role and
  token. No adapter location is guessed from `PATH`, probed, or invented.
- The command is idempotent and never overwrites operator state: rerunning
  it creates only what is missing and reports each element as created or
  already present. An existing valid configuration is preserved and
  reported even when `--from` names a different source; an existing
  `.aios/.gitignore` is left unchanged. An existing configuration that
  fails validation is a conflict error, not an overwrite.
- `--check` performs the full validation and prints every action the
  command would take, writes nothing (verified byte-for-byte on the
  target), and uses the same exit codes, mirroring the `adopt --check`
  behavior.
- After `init` on a fresh repository, the existing commands operate on it
  through `--root` unchanged: the dashboard renders its empty state and a
  hand-placed valid Task runs through the engine, with no aios-lab-specific
  project id, directory name, or hard-coded path affecting behavior.
  Repositories prepared by hand without `init` keep working exactly as
  before.
- Unit tests under `test/` cover fresh initialization, both `.git` forms,
  every fail-closed rejection with its exit code, idempotent reruns,
  preservation of existing configuration and `.gitignore`, the adapter
  portability gate for absolute, target-relative, and unresolvable tokens,
  `--check` writing nothing, and CLI wiring including usage errors.
  `npm test` passes for the whole repository.

## Constraints

- Do not create Task documents, dispatch a Worker, spend provider capacity,
  or start any server, daemon, or background process.
- Do not change the behavior of `run`, `progress`, `adopt`, or `dashboard`,
  the Loop Engine, progression, or Worker execution; changes to `src/` are
  limited to the new subcommand and its reusable helpers.
- Do not add provider SDKs, network access, credential handling, or new
  runtime dependencies, and do not copy or rewrite worker adapter source
  code into the target.
- Committed documents in the target must remain provider-neutral: nothing
  the command writes outside the machine-local configuration may contain a
  provider name requirement, user-local path, or secret.

## Context

`src/cli.js` shows the subcommand pattern, `--root` handling, and the
adopt exit-code convention; `src/workers.js` and `src/routing.js` already
validate `aios.assignments/v1` and `aios.routing/v1` and treat assignment
commands as trusted local configuration; `src/documents.js` provides the
atomic write helpers. In this repository user-local adapter paths live only
in the assignment configuration while Task, Review, and plan documents stay
provider-neutral, and `.aios/runtime/` is git-ignored; `init` reproduces
that treatment in the target via `.aios/.gitignore` because it must not
edit the target's own top-level files. The shared target-inspection
contracts define every rejection this command maps to exit codes.
