# Add a CONTRIBUTING guide for the loop workflow

## Objective

Add a `CONTRIBUTING.md` that tells a newcomer how work flows through this
repository: Tasks as the source of truth, the loop run, the three Roles,
the approval gate, and the operator's commit rhythm.

## Acceptance Criteria

- `CONTRIBUTING.md` exists at the repository root and explains, in this
  order: Task documents as the durable worklist, how to define a new Task
  (ID pattern, template, front matter ownership), how to start the loop
  with `aios run`, the three permanent Roles and how Assignments bind them
  to Workers, the `approval: required` gate and the decision-file protocol,
  and the operator's role (defining Tasks, committing after loop steps,
  resuming halted runs).
- Every statement is accurate with respect to the actual contracts under
  `.aios/` and the README; nothing is invented.
- The guide links to `.aios/tasks/README.md`, `.aios/reviews/README.md`,
  `.aios/results/README.md`, and `README.md` instead of duplicating their
  content, and stays under 120 lines of plain Markdown.
- `npm test` still passes (no code changes are expected).

## Constraints

- Documentation only: do not modify `src/`, `workers/`, `test/`,
  `package.json`, or anything under `.aios/`.
- Do not duplicate the engine documentation in `README.md`; summarize and
  link.

## Context

This Task carries `approval: required`: even after a passing review, a
human decides whether the repository takes on contributor-facing
documentation at this stage of the experiment. That decision is a policy
call, not a quality judgment.
