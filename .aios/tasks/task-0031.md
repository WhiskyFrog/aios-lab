# Define cross-repository target contracts and validation

## Objective

Add a shared, deterministic, fail-closed inspection layer that decides
whether and how AIOS may operate on a target repository, without performing
any write. Given a candidate root path, it classifies the target's
repository and `.aios/` state; given operator input, it resolves the project
id, derives or validates a plan id, and accepts or rejects a
natural-language objective. Every rejection is a distinct error naming its
exact cause, so the bootstrap and intake commands built on top of it can
fail closed before their first filesystem mutation.

## Acceptance Criteria

- A reusable module exposes a pure target classification for a candidate
  root path with exactly these outcomes: not a directory; not a repository
  root (no `.git` directory or file directly under the path); uninitialized
  (repository root with no `.aios/` entry); initialized (repository root
  whose `.aios/` state passes inspection); and conflicting, with a
  path-specific reason (for example `.aios` exists but is not a directory,
  or an existing assignment/routing configuration fails its schema
  validation). Classification never creates, modifies, or deletes anything.
- Repository-root detection checks only for a `.git` directory or file at
  the top of the target path. It does not invoke a `git` executable, walk
  parent directories, or accept a bare repository; each unsupported shape
  is reported distinctly.
- Existing assignment/routing configuration is inspected with the already
  exported `aios.assignments/v1` and `aios.routing/v1` validation rather
  than a new parser, and an inspection failure carries the underlying
  validation message.
- Project-id resolution is deterministic and documented: when the target's
  `.aios/tasks/` contains valid Task documents, their `project` value is
  reused and a conflicting explicit value is an error; otherwise an
  explicit operator-supplied project id wins; otherwise the target root's
  directory basename is accepted only when it already satisfies the
  existing `^[a-z0-9][a-z0-9-]*$` project pattern. When none of these
  applies, resolution fails with an error that names the flag the operator
  must supply. No silent normalization invents an identity.
- Plan-id handling supports both an explicit id validated against the
  existing plan-id pattern and a deterministic slug derived from the
  objective's leading words (lowercased, non-alphanumeric runs collapsed to
  single hyphens, trimmed, bounded length). An underivable slug and an
  invalid explicit id are distinct errors.
- Objective validation rejects, with distinct errors and before any caller
  writes: an objective that is empty after trimming; an objective exceeding
  a single documented size constant; and an objective containing a line
  that would parse as a markdown heading or as an attempt-frame marker,
  because such lines would corrupt the strict Task document structure when
  embedded verbatim. Accepted objectives are returned byte-for-byte
  unchanged.
- A shared error type distinguishes operator-input problems (malformed
  values the operator can correct on the command line) from target-state
  problems (the repository's condition), so command wiring can map them to
  exit codes 64 and 1 respectively, consistent with the adopt command's
  convention.
- Unit tests under `test/` cover every classification outcome, the `.git`
  file and directory forms, conflicting `.aios/` shapes, invalid existing
  configuration, each project-id resolution branch and its failure, slug
  derivation and its edge cases, every objective rejection rule including
  boundary sizes, and byte-for-byte objective preservation. `npm test`
  passes for the whole repository.

## Constraints

- Pure inspection only: no command-line flags, no subcommand wiring, no
  writes to any repository, and no changes to the Loop Engine, progression,
  adoption, dashboard, or Worker execution.
- Reuse the existing document, assignment, and routing validation exports;
  do not add alternate parsers for documents that already have one.
- Do not change any `aios.task/v1`, `aios.review/v1`, `aios.result/v1`,
  `aios.plan/v1`, `aios.assignments/v1`, or `aios.routing/v1` contract.
- No new runtime dependencies, no network access, and no invocation of
  external executables.

## Context

`src/documents.js` owns strict Task validation and the
`^[a-z0-9][a-z0-9-]*$` project pattern lives in `src/contracts.js`;
`src/plans.js` validates plan ids and shows the aggregated-problems error
style; `src/routing.js` exports the execution-configuration validation that
accepts both `aios.assignments/v1` and `aios.routing/v1`. The engine reads
`.aios/tasks/` and `.aios/reviews/`, progression and the human approver use
`.aios/approvals/`, ledgers live under `.aios/runtime/`, and Results under
`.aios/results/`; classification must treat that layout as the definition
of an initialized target. This inspection layer is the safety foundation
the bootstrap and intake commands build on.
