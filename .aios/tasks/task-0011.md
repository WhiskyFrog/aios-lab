# Extend capacity deferrals to the Codex adapter

## Objective

Close the capacity-coverage gap task-0010 exposed: a Codex usage-limit
lockout should become the same structured capacity deferral the engine
already knows how to wait through, instead of an ordinary failure that
needs an operator-scheduled resume.

## Acceptance Criteria

- The investigation is recorded: what machine-readable signal (if any)
  `codex exec --json` emits for a usage-limit rejection — event shape,
  reset time field, and session id — captured as an NDJSON fixture from
  the real CLI, not transcribed from prose.
- When and only when the Codex CLI supplies a machine-readable reset time
  and a resumable session id, `workers/codex-worker.mjs` maps a
  usage-limit rejection to an `aios.worker-execution/v1` capacity deferral
  carrying that UTC `retry_at` and an opaque continuation that resumes the
  exact session (`codex exec resume <id>`); on resume it uses that
  continuation.
- Reset times are never derived from human-readable message text, token
  counts, or hard-coded quota windows; a rejection without a
  machine-readable reset time or session id remains an ordinary failure,
  as do authentication, billing, permission, network, timeout, and
  malformed-output errors.
- If the investigation concludes the installed CLI provides no
  machine-readable reset signal at all, the adapter keeps its current
  behavior, the limitation is documented in the README Codex section with
  the captured evidence, and this criterion supersedes the mapping
  criteria above.
- Codex sessions that produce a structured execution (completed, failed,
  or deferred) are recorded in the existing `.aios/runtime/sessions.json`
  ledger with the same row semantics the Claude adapter uses, so the
  dashboard's Worker Sessions section covers both vendors.
- Behavior shared with the Claude adapter stays in
  `workers/worker-shared.mjs`; the Claude adapter's observable behavior is
  unchanged and all existing tests keep passing.
- Tests cover the new mapping (or the documented absence) with fake CLI
  fixtures and no network or billed session; `npm test` passes for the
  whole repository.

## Constraints

- Do not modify anything under `src/` or the v1 document contracts; the
  engine's deferral eligibility rules from task-0009 are fixed inputs.
- Do not modify `.aios/` documents or `.aios/assignments.json`.
- No new runtime dependencies. Do not commit or push from a Worker
  session.
- Do not upgrade or reconfigure the globally installed Codex CLI; work
  against `codex-cli 0.142.3` as found.

## Context

During task-0010, a Codex account usage-limit lockout ("try again at
4:05 AM", gpt-5.5) surfaced as an ordinary failure inside the e2e run,
halting the loop for operator-scheduled resumption — exactly the manual
trigger task-0009 was built to eliminate on the Claude side. The engine
waits only for structured deferrals with a finite future `retry_at` and
an opaque continuation (`.aios/results/README.md` Command Transport,
README Capacity Wait section, `src/sessions.js` validation).
`workers/claude-worker.mjs` shows the reference mapping from a provider's
structured rate-limit event; `codex exec` offers `--json` (JSONL events)
and a `resume` subcommand. `fixtures/claude-stream.js` and
`fixtures/codex-cli.js` show the existing fixture style.
