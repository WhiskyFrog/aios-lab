# Corroborate Codex capacity through app-server

## Objective

Use Codex's structured app-server account and thread records to turn a real
usage-limit rejection into an AIOS capacity pause, then resume the exact Codex
thread after the provider's reset time without deriving quota data from prose.

## Acceptance Criteria

- A failed `codex exec --json` turn is probed through the same Codex launcher,
  environment, and working directory using the official app-server JSONL
  handshake.
- Capacity is deferred only when the exact failed thread's latest persisted
  turn reports `codexErrorInfo: usageLimitExceeded`, the Codex account snapshot
  reports a recognized reached type, and every exhausted rate window supplies
  a finite future Unix `resetsAt`.
- Multiple exhausted windows defer until the latest reset. The deferral carries
  that UTC time and the exact `thread.started.thread_id` as its continuation.
- Missing app-server support, protocol errors, mismatched or unreadable
  threads, other structured error codes, missing or stale reset times, and
  non-window credit failures remain ordinary Worker failures.
- The probe does not parse human-readable error text, infer from token usage,
  inspect local rollout files directly, hard-code a quota duration, or consume
  an available account reset credit.
- A capacity continuation still runs `codex exec resume <thread-id>` and rejects
  a resumed thread whose reported id differs. Existing engine capacity waiting
  resumes this continuation when `--wait-for-capacity` is enabled.
- Fake-CLI tests cover app-server handshake, strict evidence selection,
  multiple windows, fail-closed behavior, deferral, and exact-thread resume.
  The full repository test suite and `git diff --check` pass.

## Constraints

- Do not change `src/`, the v1 document contracts, capacity wait policy, or
  Claude adapter behavior.
- No new runtime dependency and no Codex CLI upgrade or reconfiguration.
- Do not commit or push as part of this Task.

## Context

Task-0011 correctly established that public `codex exec --json` events expose a
thread id but no structured reset time. The installed Codex app-server protocol
adds the missing independent records: `thread/resume` exposes a structured
`TurnError.codexErrorInfo`, and `account/rateLimits/read` exposes `usedPercent`,
`rateLimitReachedType`, and Unix `resetsAt` windows. This follow-up connects
those records without weakening task-0011's prohibition on prose-derived
capacity decisions.

### Attempt 1

#### Summary

Added an isolated Codex app-server capacity client that initializes the official JSONL protocol, reloads only the exact failed thread's latest turn without item contents, and reads the authenticated Codex account rate-limit snapshot. The Codex worker now converts a failure to capacity_deferred only when the latest turn is structurally usageLimitExceeded and all exhausted windows have future structured resetsAt values; it uses the latest reset and the exact thread id. App-server launch, protocol, thread, and reset failures fall back to the original ordinary failure. Updated fixtures, Codex documentation, and tests for strict evidence, multiple windows, unavailable probes, and exact continuation resume.

#### Verification

`npm test` passed 123/123 and `git diff --check` passed. A read-only runtime probe against the installed authenticated Codex app-server confirmed `account/rateLimits/read` returns a structured Codex window and that `thread/resume` accepts `excludeTurns` with a descending one-turn `initialTurnsPage` whose item view is `notLoaded`; no thread content or reset credit was consumed.
