# Wire inter-Task progression into the CLI as `aios progress`

## Objective

Add an `aios progress <plan-dir>` subcommand to `src/cli.js` that is the
operator-invoked execution surface for the core progression library built
earlier in this plan: it drives one adopted plan's ordered Tasks forward
through the existing single-Task Loop Engine until a stop condition is
reached, and reports the current plan, current Task, completed count, stop
reason, and exact operator action in the same style as the existing `run`,
`dashboard`, and `adopt` subcommands. This proposal carries `approval:
required` because it is the concrete mechanism that automates transitions
between Tasks without a human invoking each one, the same reason task-0017
itself required approval.

## Acceptance Criteria

- `node src/cli.js progress <plan-dir> [--root <path>] [--assignments
  <path>] [--timeout-ms <n>] [--wait-for-capacity] [--max-capacity-wait-ms
  <n>] [--max-capacity-pauses <n>]` is accepted with the same option
  parsing, defaults, and validation style `run` already uses for the options
  they share; `<plan-dir>` follows the same resolution rule `adopt` already
  uses (absolute path used as-is, relative path resolved against `--root`).
- The command locates and loads the plan's ordered real Task ids using only
  the core library from earlier in this plan; it does not re-implement
  `PLAN.md` parsing, Task loading, or engine invocation.
- On success the command prints one JSON report naming: the plan id, the
  ordered list of completed Task ids, whether the whole plan is now
  complete, the id of the Task it stopped on (or null when complete), the
  stop reason, and the exact operator action string from the core library
  verbatim.
- This proposal fixes `progress`'s exit codes itself rather than leaving the
  mapping to its Implementer; they are documented in `--help`/usage text in
  the same style as `run`'s existing line and are exactly:
  - `0` — every Task in the plan reached `done` (plan complete).
  - `3` — stopped awaiting human approval (core library reason `awaiting
    approval`, engine `category: "approval_gate"`).
  - `4` — stopped on a blocked Task (rejected or retry-limit-exhausted).
  - `5` — stopped on a capacity wait (`kind: "waiting"`).
  - `6` — stopped on cancellation (`category: "cancelled"`).
  - `7` — stopped on any other halt (`category: "worker_failure"`,
    `"invalid_document"`, or `"conflict"`).
  - `64` — command usage error.

  `0` and `64` intentionally carry the same meaning `run`, `dashboard`, and
  `adopt` already give those two codes (success and usage error); `3`, `4`,
  `5`, `6`, and `7` are new codes chosen specifically to avoid colliding with
  `run`'s existing `1` (halted), `2` (blocked), and `75` (waiting), so a
  script that branches on both `aios run` and `aios progress` exit codes
  never confuses one command's code for the other's.
- `--wait-for-capacity`, `--max-capacity-wait-ms`, and
  `--max-capacity-pauses` are passed through to each underlying
  `LoopEngine.run` invocation unchanged from their existing `run` semantics.
- Cancellation (SIGINT/SIGTERM) is wired through an `AbortSignal` covering
  the whole multi-Task loop, not only a capacity-wait sleep, consistent with
  the cancellation classification added to the core library; a cancelled
  command reports the completed-so-far list and stops without starting a
  new Task.
- Re-running the exact same command after any stop reason, with no other
  repository change, resumes from repository state: already-`done` Tasks
  are not rerun, and the command reaches the same next stop point
  deterministically (idempotent re-invocation), consistent with the core
  library's guarantee.
- Automated tests cover: successful multi-Task completion from one command
  invocation; each documented exit code reached through a fixture
  Task/Review/Assignment setup; option parsing and validation errors;
  cancellation mid-run; and idempotent re-invocation through the CLI entry
  point itself (not only the core library).
- Automated regression tests prove the existing `run`, `dashboard`, and
  `adopt` subcommands' argument parsing, output, and exit codes are
  unchanged by this addition — run the full existing `cli.test.js` suite
  and add any missing regression coverage for shared parsing code paths.
- `--help`/usage output documents the new subcommand alongside the existing
  three, in the same format.

## Constraints

- Do not change the argument parsing, output, or exit-code behavior of the
  existing `run`, `dashboard`, or `adopt` subcommands.
- Do not duplicate Role dispatch, Review projection, Approval handling,
  capacity continuation, retry limits, or session recording in `src/cli.js`;
  all of that stays inside `LoopEngine`, invoked only through the core
  progression library.
- No background daemon, watcher, or implicit auto-start; `aios progress`
  only acts when explicitly invoked by an operator, exactly once per
  invocation's worth of work.
- Add no new runtime dependency.

## Context

`src/cli.js` already implements `run`, `dashboard`, and `adopt` with shared
conventions: `--root`-relative path resolution, one JSON report line on
stdout, and documented exit codes in `usage()`. The core progression library
delivered earlier in this plan exposes the ordering, resume, and stop-policy
logic this subcommand must call; this proposal's own scope is argument
parsing, operator-facing reporting, exit codes, and signal wiring, mirroring
how thin the existing `run` subcommand already is over `LoopEngine`.
