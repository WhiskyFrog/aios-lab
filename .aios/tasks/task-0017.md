# Plan automatic progression across adopted Tasks

## Objective

Produce a reviewed `software-feature` plan under
`plans/inter-task-progression/` that lets an operator start or resume an
adopted plan and have AIOS advance through its ordered Tasks without manually
invoking each Task, while preserving every Review, Approval, capacity, and
failure gate.

## Acceptance Criteria

- `PLAN.md` selects the `software-feature` profile and explains how contracts,
  execution state, recovery behavior, CLI integration, and end-to-end
  verification shape the decomposition.
- The plan defines the smallest operator-invoked execution surface that reads
  the real Task order already written into an adopted `PLAN.md`; it does not
  introduce Task dependency fields or require a daemon.
- Automatic progression invokes the existing single-Task Loop Engine rather
  than duplicating Role dispatch, Review projection, Approval handling,
  capacity continuation, retry limits, or session recording.
- Progression is deterministic and resumable: completed Tasks are not rerun,
  the next unfinished Task is selected from plan order, and invoking the
  command again after a stop continues from repository state rather than from
  hidden in-memory state.
- The proposed stop policy is explicit and testable. AIOS must stop at a human
  approval gate, blocked Task, Worker failure, invalid document, exhausted
  retry limit, capacity wait that cannot be completed in the current process,
  cancellation, or repository mutation conflict; it must never skip a failed
  or undecidable Task to run a later one.
- Human control remains visible: the command reports the current plan, current
  Task, completed count, stop reason, and exact operator action required. The
  plan also covers how the dashboard should expose an active/stopped plan run
  without turning the dashboard into a mutating control surface.
- The decomposition keeps implementation, recovery/stop semantics,
  operational visibility, and end-to-end verification in focused Tasks where
  that separation improves one-session reviewability. Execution order lives
  only in `PLAN.md`.
- The plan requires unit coverage for ordering, resume, every stop condition,
  idempotent re-invocation, and unchanged existing `aios run` behavior, plus a
  disposable end-to-end demonstration in which two adopted Tasks complete in
  order from one operator command.
- `node src/cli.js adopt plans/inter-task-progression --check` passes without
  writing Tasks, and the Implementer's Verification records that command and
  the focused plan-validation result.

## Constraints

- This is a planning Task. The Implementer writes only under
  `plans/inter-task-progression/`; it does not implement the runner or modify
  `.aios/`, `src/`, `test/`, workers, assignments, or git history.
- Do not weaken or bypass Review and Approval gates. Do not add a background
  daemon, watcher, concurrent scheduler, or implicit auto-start on adoption.
- Prefer repository documents as the source of truth. Any proposed new
  persisted execution state or contract must be justified against deriving
  progress from the adopted plan and existing Task states.
- Keep adaptive model selection out of this plan so automatic progression can
  be proven independently with the current Assignment mechanism.
- This Task carries `approval: required` because the resulting plan will extend
  the engine's execution surface and automate transitions between Tasks.

## Context

The Planner experiment successfully turned the dashboard brief into an
approved plan and adopted `task-0015` and `task-0016` in the intended order.
However, each adopted Task still required a separate operator trigger, and the
second Task's Claude Implementer timed out twice before a root agent manually
completed and verified the retained work. The next engine milestone is
therefore not broad concurrency: it is a narrow, observable, resumable way to
advance one adopted plan sequentially while stopping safely whenever human or
Worker intervention is required.

## Follow-on Goal

After inter-Task progression is stable, plan adaptive model routing as a
separate engine capability. For each Task and Role, a policy should evaluate
the Planner profile, complexity, risk, context size, tool needs, verification
burden, latency, and cost budget, then choose from an explicit candidate pool
that includes both Claude and Codex instead of treating either provider as the
universal default. Selection remains fitness-first, but materially equivalent
candidates should follow a configurable distribution target so work does not
silently concentrate on one provider. Where eligible, Implementer and Reviewer
should use different providers to reduce correlated blind spots. Capacity,
timeout, or provider failure should use a declared cross-provider fallback
rather than an ad hoc manual handoff. This requires evolving the current
Role-only `aios.assignments/v1` resolution into auditable Task-and-Role-aware
routing without adding provider choices to Task documents themselves.
Operators retain an explicit override, and every decision records the chosen
provider/model, considered alternatives, selection reason, fallback reason,
and distribution-policy effect. This goal must not be folded into automatic
progression because routing quality and workflow correctness need independent
evaluation.
