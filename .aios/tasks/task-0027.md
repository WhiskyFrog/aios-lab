# Add validated routing overrides to run and progress

## Objective

Give the operator an explicit, auditable way to displace an adaptive policy
choice for an exact Task/Role or a Role across the current foreground command,
without editing Task documents. Extend `aios run` and `aios progress` with one
consistent override syntax, validate every candidate and safety invariant before
launch, pass the override through the routed resolver, and report the applied
choice in command output while leaving legacy Assignment commands unchanged.

This Task carries `approval: required` because overrides directly control paid
execution resources and extend the public CLI surface used by both single-Task
and automatic sequential plan runs.

## Acceptance Criteria

- `run` and `progress` accept repeatable `--route-override
  <task-selector>:<role>=<candidate-id>`. `<task-selector>` is an exact
  `task-####` or `*`; Role is `implementer` or `reviewer`; candidate id must
  exist in the active `aios.routing/v1` catalog. `approver` remains outside
  model routing. Help and usage text document quoting on PowerShell and POSIX
  shells.
- Exact Task selectors take precedence over `*` for the same Role. Duplicate
  selectors, conflicting exact rules, malformed ids/Roles, an override supplied
  with legacy `aios.assignments/v1`, or a candidate ineligible for the selected
  Role fail with exit code 64 before any Worker launch or Task mutation.
- The routing config's optional operator-override table uses the same selector
  grammar and validation. A CLI override takes precedence over a configured
  override for the exact selector; precedence and the displaced configured
  candidate are included in audit evidence.
- An override may displace fitness, cost, latency, or provider-distribution
  preference but cannot bypass hard safety gates: enabled state, declared Role
  and tool/context capability, Planner/unknown high-tier minimum, Reviewer
  equal-or-higher tier, and bounded fallback limits. An unsafe override fails
  closed and names each violated invariant; there is no force flag.
- The override is part of the decision key input and is persisted before
  dispatch. Audit records its source (`cli` or `config`), selector, chosen
  candidate, normal policy winner, displaced rationale, and confirmation that
  hard gates passed. Re-running the same command/context reuses that decision;
  changing the override creates a policy-input conflict requiring a new explicit
  action rather than rewriting the old row.
- Fallback from an overridden candidate follows the configured bounded route
  only when the override permits fallback by policy; otherwise the command
  returns the existing stop outcome after that candidate fails. The command
  output and audit row state whether fallback was allowed. No fallback silently
  cancels the fact that the initial choice was overridden.
- `run`'s JSON report gains an optional sanitized `routing` summary for the Role
  action most recently dispatched: candidate/provider/model/tier, override
  source, step, and fallback/escalation reason. `progress` reports the same
  summary for the Task on which it stopped, or null when no routed action ran.
  Existing fields and exit-code meanings are unchanged.
- With no override flags and a legacy Assignment file, stdout is byte-for-byte
  unchanged. With routing enabled but no override, behavior is the policy choice
  and the optional routing summary is populated. Existing engine/capacity flags
  preserve their parsing, defaults, and semantics.
- Tests cover syntax and precedence, wildcard versus exact selectors, configured
  versus CLI override, every hard-gate rejection, legacy-config rejection,
  fallback allowed/denied, decision idempotency/conflict, JSON reports, all
  existing exit codes, and unchanged no-override legacy `run` and `progress`.

## Constraints

- Do not add provider/model fields to Task or Plan documents and do not accept
  raw command argv, executable paths, credentials, or arbitrary environment
  variables from CLI flags. Overrides name only validated catalog candidate ids.
- Do not add an override that can weaken Planner tier, Reviewer tier, required
  capabilities, Review/Approval gates, Task retries, or recovery bounds.
- Keep CLI handling thin over the routing resolver. Do not duplicate selection,
  ledger, fallback, or Task-order logic in `src/cli.js`.
- Do not add dashboard UI, interactive prompts, concurrency, a daemon, or a
  provider SDK in this Task.

## Context

`src/cli.js` currently shares engine option parsing between `run` and
`progress`, including Assignment path, timeout, and capacity settings. An
override should follow that same shared path and name a safe catalog id rather
than injecting a new command. The human Approver remains bound through the
existing Assignment mechanism and is intentionally not model-routed.
