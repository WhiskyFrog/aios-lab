# Implement aios brief objective intake

## Objective

Add an `aios brief` subcommand that turns one operator objective into
exactly one valid planning Task in the target repository's `.aios/tasks/`.
The generated document carries the objective verbatim as its brief and
scopes its Implementer to producing `plans/<plan-id>/` in that same
repository under the existing Planner protocol and profile table. The
command creates that single document and nothing else: it never dispatches
a Worker, spends provider capacity, or modifies existing Tasks, and the
generated wording grants the Planner no authority beyond the existing
convention.

## Acceptance Criteria

- `aios brief <objective> [--root <path>] [--plan <plan-id>]
  [--project <project-id>] [--check]` joins the existing CLI with usage
  text, `--root` defaulting to the current working directory, and exit
  codes 0 (created or `--check` passed), 1 (target validation failure), and
  64 (usage error), consistent with the adopt command's convention.
- All validation happens through the shared target-inspection contracts
  before any write, each failure producing its distinct error and leaving
  the target untouched: a target that is not a repository root or not
  initialized (the error names the bootstrap step to run first); an empty,
  oversized, or structurally unsafe objective; an unresolvable project id;
  an invalid or underivable plan id; a `plans/<plan-id>/` directory that
  already exists in the target; and a Task id collision.
- The Task id is the next sequential id after the greatest existing id in
  the target's `.aios/tasks/`, reusing the existing allocation logic, and
  the document is written atomically with an exclusive create so a
  concurrent or stale collision fails closed rather than overwriting.
- The generated front matter is exactly `schema: aios.task/v1`, the
  allocated id, the resolved project, a bounded single-line title derived
  deterministically from the objective, `state: implement`,
  `retry: {count: 0, limit: 2}`, `approval: required`, and
  `last_review: null`. No flag relaxes the approval default in this
  version.
- The generated body contains: an Objective directing the Implementer to
  produce a reviewed plan under `plans/<plan-id>/` in the same repository
  following the Planner protocol and choosing from the documented profile
  table; a Brief section carrying the operator objective byte-for-byte;
  Acceptance Criteria requiring valid `aios.plan/v1` front matter with a
  registered profile and non-empty profile reason, the required plan
  sections, contiguous valid proposals with ordering only in the plan
  document, and an Implementer Verification that records a passing
  non-mutating adopt check of `plans/<plan-id>/`; Constraints limiting
  writes to `plans/<plan-id>/`, forbidding `.aios/` modification and
  Worker dispatch, and stating that proposals become Tasks only through
  operator-invoked adoption; and an empty Attempts section.
- The template is provider-neutral and target-neutral: no provider or model
  names, no user-local paths or secrets, and no aios-lab-specific project
  id, directory name, or hard-coded path. A test loads the generated
  document through the engine's strict Task validation from a
  non-aios-lab-named scratch repository.
- The command performs exactly one filesystem write on success and reports
  the allocated Task id, plan id, and project id as structured output.
  `--check` reports the same resolution without writing anything, verified
  byte-for-byte on the target.
- `run --root` accepts the generated Task immediately: an engine-level test
  (with a stubbed assignment) reaches the Implementer Role on it without
  any manual edit, and existing `run`, `adopt`, `progress`, and `dashboard`
  behavior is unchanged by regression.
- Unit tests under `test/` cover id allocation and continuation after
  existing Tasks, exclusive-create collision handling, verbatim brief
  preservation including multi-line objectives, every fail-closed
  rejection with its exit code, project and plan id resolution branches,
  `--check`, and CLI wiring. `npm test` passes for the whole repository.

## Constraints

- The command writes exactly one new file under the target's
  `.aios/tasks/` and nothing else; it never modifies existing Tasks,
  Reviews, plans, configuration, or ledgers, and never deletes anything.
- Do not dispatch a Worker, spend provider capacity, or start any server,
  daemon, or background process; materializing the Task must not trigger
  the engine.
- Do not change any document contract, the Loop Engine, progression,
  adoption, or dashboard behavior; changes to `src/` are limited to the new
  subcommand and its reusable helpers. The generated wording must not
  extend Planner authority: adoption remains a separate operator-invoked
  command.
- No provider SDKs, network access, or new runtime dependencies.

## Context

The Planner convention (README "Planner" section) defines the protocol,
profile table, and the requirement that a planning Implementer verify with
a non-mutating adopt check; `src/plans.js` exports `allocateTaskIds` and
the plan-id rules; `src/documents.js` exports strict Task validation, the
required Objective and Acceptance Criteria body sections, and atomic write
helpers; `src/cli.js` shows the subcommand and exit-code pattern. The
shared target-inspection contracts supply repository, initialization,
project-id, plan-id, and objective validation, and the bootstrap command
prepares the `.aios/` state this command requires. Today the operator
hand-writes a planning Task in the target repository; this command replaces
that step with a validated, repeatable materialization.
