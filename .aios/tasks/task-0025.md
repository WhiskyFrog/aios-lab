# Select models deterministically and record routing decisions

## Objective

Implement a pure, explainable routing policy over a validated candidate catalog
and normalized workload context, plus an atomic routing-decision ledger. The
policy applies capability and safety gates before cost or provider distribution,
keeps Planner work on a high tier, allows lower-tier implementation only when
the workload evidence explicitly qualifies, prefers an equal-or-stronger
different-provider Reviewer, and makes the first choice idempotent for the same
Task/Role action.

## Acceptance Criteria

- The selector accepts only validated routing configuration, normalized
  workload context, an exact routing decision key, prior decision history, and
  an optional Implementer decision for Reviewer comparison. It performs no file
  I/O, Worker launch, clock read, random draw, or model call.
- Selection occurs in this fixed order: remove disabled/Role-ineligible
  candidates; enforce required context/tool capabilities and policy budgets;
  enforce the workload minimum tier and Role safety invariants; compute a
  documented lexicographic fitness tuple; retain only the best materially
  equivalent tuple; apply provider distribution deficit within that set; then
  break a remaining tie by stable candidate id. Cost or distribution can never
  restore a candidate removed by a capability, risk, or tier gate.
- Planning always requires a candidate at or above the configured high tier.
  No cost budget, latency preference, provider deficit, or override placeholder
  in this policy lowers that minimum. Unknown or conservative workload context
  receives the same protection.
- A lower-tier candidate is eligible only for an Implementer context whose
  normalized lower-tier gates all passed. Reviewer selection requires a tier
  equal to or higher than the Implementer decision for the same Task attempt;
  if that decision is unavailable, Reviewer minimum tier is conservative high.
- When eligible, Reviewer fitness prefers a provider different from the
  Implementer's. Same-provider review is allowed only when no different-provider
  candidate survives all higher-priority gates, and the exact disqualifying
  reasons for cross-provider alternatives are recorded.
- "Materially equivalent" is mechanical: candidates have the same best
  capability/tier/latency/cost fitness tuple after hard gates. Provider target
  weights operate only inside that set. The distribution rule compares each
  provider's actual decisions in the configured finite window with its weighted
  expected share, chooses the greatest deficit, and uses provider id then
  candidate id as stable tie breakers. Tests use integer/rational arithmetic or
  another representation that cannot drift due to floating-point rounding.
- The decision key contains Task id, Role, Task attempt/review generation, and
  routing policy revision. The policy exposes an explicit next-step index for
  bounded fallback/escalation. The key does not use wall-clock time, session id,
  mutable distribution count, or provider continuation.
- A strict `aios.routing-decisions/v1` ledger is stored atomically at
  `.aios/runtime/routing-decisions.json`. Each immutable decision/step records:
  key and policy revision; normalized workload summary and evidence sources;
  ordered considered candidates with eligible/ineligible reason codes; chosen
  candidate/provider/model/tier; fitness tuple; distribution window counts,
  target, deficit, and whether distribution changed the winner; override data
  when supplied later; parent/step relationship; fallback or escalation reason;
  dispatch/outcome state; and timestamps supplied by the caller.
- The ledger never stores command argv, environment variables, credentials,
  prompt/Task bodies, continuation tokens, or raw provider errors. Errors are
  normalized to a small reason-code vocabulary with a separately bounded,
  sanitized diagnostic string.
- Persisting the initial decision is compare-and-swap safe and occurs before
  dispatch. Resolving an existing key with the same policy revision returns its
  recorded active candidate even when newer ledger rows changed distribution.
  A mismatched immutable field or policy revision fails closed; it never
  silently rewrites history or takes a fresh draw.
- The ledger provides read APIs for the latest Task/Role decision, the exact
  key, bounded distribution-window counts, and a sanitized dashboard projection.
  A missing ledger is an empty state; malformed or conflicting content is a
  named error and is never overwritten automatically.
- Unit tests cover hard-gate precedence, Planner high-tier enforcement,
  lower-tier eligibility and every rejection reason, Reviewer tier/provider
  rules, weighted distribution convergence and tie cases, same-key idempotency,
  policy-revision mismatch, atomic conflict handling, malformed ledger failure,
  sanitized storage, and proof that distribution never beats fitness.

## Constraints

- Keep candidate selection pure and independently testable. File persistence
  belongs only to the ledger adapter around it.
- Do not launch or retry Workers, change Task state, increment Task retries,
  modify `LoopEngine`, parse CLI overrides, or render the dashboard in this
  Task.
- Do not weaken conservative workload defaults. There is no "cheap model"
  shortcut and no random, opaque, or AI-generated routing score.
- Use repository-local atomic replace/conflict patterns already established for
  durable AIOS files; do not add a database, lock service, daemon, or dependency.

## Context

The session ledger records what a Worker session consumed and whether it
completed, failed, or deferred, but it does not capture candidates rejected
before launch or why distribution affected a choice. A separate decision ledger
is therefore required. The selector is deliberately separated from dispatch so
its fitness-first and idempotency properties can be proven without live Claude
or Codex capacity.
