# Implement automatic Loop Engine dispatch

## Objective

Implement a single-process Loop Engine that needs one operator trigger per
Task, then automatically dispatches each next Role to its assigned Worker
until the Task reaches `done`, `blocked`, approval, or an execution halt.

## Acceptance Criteria

- The engine derives the next permanent Role only from Task state and resolves
  that Role through the current Assignment before every action; engine control
  flow contains no worker-specific behavior.
- Every Worker implements one `execute(Task) -> Result` interface, and changing
  a Role's Assignment does not require changing the engine.
- A single run automatically performs the Implementer-to-Reviewer loop,
  including `changes_requested` retries and retry exhaustion, without an
  operator triggering each Role separately.
- Successful Implementer, Reviewer, and Approver Results are projected
  according to the existing Task, Review, and Result contracts, including
  atomic Task rewrites and immutable Review creation.
- Before invoking a Reviewer, the engine applies the documented orphan Review
  recovery rule.
- Worker failures, invalid Results, and missing Assignments halt the current
  run without changing Task state or consuming a retry.
- The engine re-reads the Task after Worker execution and halts on a concurrent
  Task change instead of overwriting machine-owned state.
- A generic command Worker can receive the Task document and return one
  `aios.result/v1` JSON object, allowing external workers to be replaced by
  configuration.
- An operator can start the loop for a Task from one command and receive a
  clear terminal or halted outcome.
- Automated tests cover direct pass, changes followed by pass, retry
  exhaustion, worker failure, orphan Review recovery, optional approval, and
  Assignment replacement.

## Constraints

- Do not add Redis, a message queue, an event bus, a server, a scheduler, or a
  background daemon.
- Support one Loop Engine process; distributed locking and exactly-once worker
  execution remain outside Sprint 0.
- Do not add provider-specific Claude, Codex, Gemini, or API behavior to the
  engine.
- Do not persist Result envelopes or add event history.
- Repository changes made by an external Worker are not rolled back when that
  Worker fails; the engine guarantees unchanged Task and Review control state,
  then requires an explicit resume.
- Do not modify completed Tasks, existing Reviews, or the v1 document
  contracts.
- Use the filesystem repository as the only durable store.

## Context

Tasks 1 and 2 established the Task, Review, and Result contracts. Today the
operator manually starts the Implementer and Reviewer in separate sessions.
This Task replaces those repeated manual triggers with a synchronous,
state-driven dispatcher. A queue is intentionally deferred until work must be
scheduled concurrently or across processes.
