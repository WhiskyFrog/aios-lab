# Harden the Claude worker adapter's failure and reply parsing

## Objective

Resolve the two non-blocking notes recorded in `review-0006` so that
`workers/claude-worker.mjs` fails along exactly one path and parses session
replies deterministically, with the parsing logic under automated test.

## Acceptance Criteria

- No code path in `workers/claude-worker.mjs` can continue past a failure:
  after a failure is reported, the adapter cannot report a second failure or
  write anything to stdout. Failure call sites make this structurally
  evident (for example, the failure helper never returns and call sites
  return or throw), not merely coincidental.
- Reply parsing extracts the payload object correctly when the JSON object
  is the entire reply, is wrapped in markdown code fences, or is surrounded
  by prose — including prose that itself contains unrelated braces. A reply
  from which no payload can be deterministically extracted is treated as
  unusable. The chosen extraction rule is documented in a comment.
- The reply parsing and payload validation logic is exported and covered by
  a `node --test` suite under `test/` that runs via `npm test` without
  launching any Claude session or network access.
- The adapter's observable contract is unchanged: same Assignment argv
  shape, same stdin/env inputs, same single `aios.result/v1` object on
  stdout for usable replies, `status: failure` for a `failure_reason`
  reply, and a nonzero exit with a stderr diagnostic for unusable output.
- `npm test` passes for the whole repository.

## Constraints

- Do not modify anything under `src/` or the v1 document contracts.
- No new runtime dependencies.
- Keep `workers/claude-worker.mjs` a single self-contained script runnable
  exactly as today; exporting functions for tests is allowed.
- Do not modify `.aios/` documents or `.aios/assignments.json`.

## Context

`review-0006` (in `.aios/reviews/`) passed task-0004 with two non-blocking
notes: (1) several `fail()` call sites are not followed by a return, so
some paths could in principle report twice; (2) `extractPayload` uses a
first-`{` to last-`}` slice, which misparses replies containing stray
braces outside the intended JSON object. The adapter is
`workers/claude-worker.mjs`; existing tests live in `test/*.test.js` and
run with `node --test` on Node 22.

### Attempt 1

#### Summary

Hardened workers/claude-worker.mjs to resolve both review-0006 notes. (1) Replaced fail()'s process.exit(1) with a WorkerFailure throw: the function now provably never returns (JS throw semantics, not convention), and all adapter logic moved into an async main() that runs only when the file is the process entry point (guarded via fileURLToPath(import.meta.url) === path.resolve(process.argv[1])), so a single top-level try/catch is the only place that writes the stderr diagnostic and sets a nonzero exit code — no path can report a second failure or reach the stdout write after one. (2) Rewrote extractPayload to scan for '{' characters, find each one's matching '}' via a string-literal-aware brace-depth scanner (skipping brace matching inside quoted strings), attempt JSON.parse on each non-overlapping span, and return the parsed object only if exactly one span parses as a JSON object — otherwise the reply is unusable (null). This deterministically handles a reply that is the whole payload, one wrapped in markdown code fences (fences contain no braces so they're skipped as prose), and one surrounded by prose with stray non-JSON braces, while refusing ambiguous replies with multiple valid JSON objects. The rule is documented in a comment above extractPayload. Exported rolePrompt, sessionArguments, extractPayload, isObject, nonEmptyString, and validatePayload for testing; added test/claude-worker.test.js (17 new node:test cases) covering all three extraction shapes, nested/string braces, ambiguous and unterminated replies, and payload validation for all three roles plus failure_reason. The Assignment argv shape, env/stdin inputs, and stdout envelope contract are unchanged.

#### Verification

Ran `node --check workers/claude-worker.mjs` (syntax OK) and `npm test` — full suite passes 41/41 (24 pre-existing + 17 new). Manually smoke-tested the CLI entry point directly (node workers/claude-worker.mjs) for three failure scenarios not requiring a live Claude session — session spawn failure (ENOENT), empty stdin, and unsupported role — and confirmed each writes exactly one stderr line, writes nothing to stdout, and exits 1. Confirmed via `git status`/`git diff --stat` that only workers/claude-worker.mjs was modified and test/claude-worker.test.js was added; no src/ or .aios/ files touched, no new dependencies added to package.json.
