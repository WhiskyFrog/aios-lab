# Establish the Planner convention and plan adoption

## Objective

Let one ordinary loop Task decompose a written product brief into a
reviewed set of proposed Task documents, and give the engine one command
that adopts a passing plan into real Tasks — without changing any v1
contract and without letting Workers write into `.aios/`.

## Acceptance Criteria

- A Planner convention is documented in the README and enforced by
  validation: a planning Task is an ordinary `aios.task/v1` Task whose
  Implementer produces, under `plans/<plan-id>/`, one `PLAN.md` (the
  brief, the decomposition rationale, and an ordered execution list that
  references proposals by placeholder) plus one proposal file per future
  Task, named `P-01.md`, `P-02.md`, and so on.
- Each proposal validates under the exact `aios.task/v1` document rules
  with exactly one relaxation: its `id` is its placeholder matching
  `^P-[0-9]{2,}$` and its filename is that placeholder plus `.md`. State
  must be `implement`, `retry` at defaults, `last_review` null. Proposal
  bodies must not reference other proposals; ordering and relationships
  live only in `PLAN.md`.
- `aios adopt <plan-dir> [--root <path>] [--check]` strictly validates the
  plan: `PLAN.md` present, every proposal valid under the relaxed rule,
  and every placeholder referenced in `PLAN.md` resolvable to a proposal
  file. `--check` validates and reports without writing anything.
- Adoption is all-or-nothing: any invalid proposal means nothing is
  written and the command exits nonzero naming every problem. On success
  it allocates sequential real Task IDs continuing the repository's
  numbering, writes each adopted Task atomically into `.aios/tasks/` with
  the real id substituted and the body otherwise preserved, rewrites the
  placeholder references in `PLAN.md` to the real IDs, and prints the
  placeholder-to-id mapping. Exit codes: 0 adopted (or `--check` passed),
  64 usage error, 1 validation failure.
- The planning-Task convention requires the Implementer's Verification to
  include a passing `aios adopt --check` of its own plan directory.
- `run` and `dashboard` behavior is unchanged; adopted Tasks appear on the
  dashboard through the existing scan with no dashboard change.
- An end-to-end sandbox demonstration: in a disposable repository, a real
  worker-backed planning Task takes a small written brief (a two-page
  website is enough) and reaches `done` through the normal loop — the
  Reviewer judges both proposal validity and whether the decomposition
  covers the brief at a one-session-per-Task grain — and `aios adopt`
  then creates the real Tasks, every one of which loads under the
  engine's strict validation. Executing the adopted Tasks is out of
  scope.
- Unit tests under `test/` cover relaxed proposal validation, ID
  allocation and continuation, all-or-nothing failure, `PLAN.md`
  reference rewriting, `--check`, and CLI wiring; `npm test` passes for
  the whole repository.
- README gains a "Planner" section: the convention, `aios adopt` usage,
  and the boundary — Workers propose under `plans/`, only the engine
  writes `.aios/`, and adoption is operator-invoked.

## Constraints

- Do not change `aios.task/v1`, `aios.review/v1`, `aios.result/v1`, or
  the worker-execution transport. No dependency fields on Tasks; ordering
  lives in `PLAN.md` prose only.
- No multi-Task scheduler, no `run --all`, no daemon; adoption is a
  one-shot operator-invoked command.
- `src/` changes are limited to the new subcommand plus reusable
  validation helpers; `run` and `dashboard` are untouched in behavior.
- No new runtime dependencies. Do not commit or push from a Worker
  session.
- This Task carries `approval: required`: operator policy since the
  task-0008 retrospective is that changes extending the engine surface
  get a human gate after review.

## Context

Sprint 1 opens here. Sprint 0 proved the loop on single Tasks
(task-0001 through task-0011); building anything larger — the standing
goal is a complete website from one brief — needs the decomposition step
itself inside the loop. Reference points: `src/cli.js` shows the
subcommand pattern (`parseDashboardArguments`), `src/documents.js`
exports the strict validation and the atomic/immutable write helpers,
and `TaskStore.nextReviewId` shows the ID-allocation pattern to mirror
for Task IDs. `plans/` sits outside `.aios/`, so Worker sessions may
write there under the existing HARD_RULES. The Reviewer Role's read-only
session can judge proposals by reading them; mechanical validation is
the job of `aios adopt --check` and the unit tests.
