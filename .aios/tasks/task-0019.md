# Build the core inter-Task progression library

## Objective

Add an internal library that, given an adopted plan directory, derives the
ordered real Task ids already written into its `PLAN.md`, then repeatedly
determines the next unfinished Task from repository state alone and
advances it by invoking the existing single-Task `LoopEngine` exactly once,
continuing to the next unfinished Task while each outcome is `done` and
stopping immediately on any other outcome, and reports a deterministic stop
outcome for the whole call — with no new persisted execution-run document,
no Task dependency fields, no daemon, and no duplication of Role dispatch,
Review projection, Approval handling, capacity continuation, retry limits,
or session recording, all of which stay inside `LoopEngine`. This proposal
delivers the library only; CLI and dashboard integration are separate
proposals in this plan.

This proposal carries `approval: required` because it is the one that
changes `LoopEngine`'s returned outcome contract, which extends the engine
surface — the same operator policy from the task-0008 retrospective, already
applied to task-0012 and task-0017, that requires a human gate after review
for engine-surface changes. That policy applies to this proposal directly,
not only to the later CLI proposal that merely consumes the extended
contract.

## Acceptance Criteria

- A function reads an already-adopted plan directory's `PLAN.md` (reusing
  the existing document-parsing and Execution Order section helpers rather
  than re-implementing Markdown parsing) and returns the ordered list of real
  Task ids (`task-####`) referenced in its Execution Order section, in
  document order.
- That function rejects a plan whose `PLAN.md` still contains any `P-##`
  placeholder (not yet adopted) with a named error, and rejects any
  referenced Task whose `project` does not equal the plan's `project`,
  without partially reporting a mixed-validity order.
- A second function selects the next unfinished Task from an order: the
  first Task whose current `state` is not `done`. Every Task at or before it
  in the order that is already `done` is treated as completed and is never
  passed to the Loop Engine. If every Task in the order is `done`, the
  function reports plan completion with the full completed list and no next
  Task.
- The progression runner's main entry point advances a plan across
  potentially many Tasks in one call: it repeats "select the next unfinished
  Task, invoke `LoopEngine.run` exactly once for it, inspect the outcome" —
  when the outcome is `done`, that Task is appended to the completed list and
  the loop immediately repeats for the new next unfinished Task; on any
  other outcome, the loop stops immediately and returns. Each Task in the
  order is passed to the engine at most once per call, strictly in order,
  and the loop never advances past a Task whose outcome was not `done`.
- `LoopEngine`'s outcome gains one new field, `category`, present on every
  outcome whose `kind` is `"halted"` and absent on every other `kind`
  (`"done"`, `"blocked"`, and `"waiting"` stay exactly as self-describing as
  they are today; the rejected-vs-retry-exhausted split within `"blocked"`
  continues to be read directly from the Task's own `approval` field versus
  its `retry.count === retry.limit`, not from a new engine field). The
  engine never returns a `"halted"` outcome without a `category`, and
  `category` is always exactly one of these five stable string values, none
  of which is ever renamed, removed, or reused for a different meaning once
  shipped: `"approval_gate"` (the halt originated from the `approver` Role
  finding no decision file yet, or an invalid decision file, at
  `.aios/approvals/<task-id>`), `"worker_failure"` (an implementer or
  reviewer Worker failed, errored, timed out, or returned an invalid
  Result), `"invalid_document"` (the Task or a Review referenced by it
  failed to load or validate), `"conflict"` (the Task document changed
  underneath the engine between the read that started this run and a
  later write), and `"cancelled"` (the run was aborted through the existing
  `AbortSignal`, whether during active Worker execution or a capacity-wait
  sleep — extending today's sleep-only cancellation detection to cover
  active Worker execution too). Backward-compatibility boundary: the
  existing `kind`, `task`, `reason`, and `retryAt` fields, their presence
  rules, and their meaning are byte-for-byte unchanged; a caller that
  destructures only those fields observes no behavior change; `category`'s
  five values may only be extended by a later addition, never by renaming or
  removing one of these five, and any such later addition is itself an
  engine-surface change that carries `approval: required` under the same
  policy that governs this proposal.
- Full cancellation coverage: a cancellation requested through the existing
  `AbortSignal` mechanism is classified as `cancelled` whether it interrupts
  an active Worker execution or a capacity-wait sleep (today only the sleep
  path distinguishes cancellation from an ordinary wait failure).
- The progression runner maps every engine outcome to exactly one of these
  stop reasons and never invents a ninth: plan complete (not a stop — all
  Tasks done), awaiting approval, blocked (rejected), blocked (retry limit
  exhausted), Worker failure, invalid document, capacity wait, cancelled, or
  repository mutation conflict. It never skips a Task in any of these states
  to attempt a later Task in the order.
- The runner's return value reports: the plan id, the id of the Task it
  stopped on (or null when the plan is complete), the ordered list of
  completed Task ids so far, the stop reason, and a short human-readable
  operator action string naming the exact next step (for example, the
  approval decision file path and its two accepted contents, or the command
  to rerun).
- Automated tests cover: correct ordering derivation from a multi-Task
  adopted plan; a not-yet-adopted plan is rejected; a Task/plan project
  mismatch is rejected; resuming a plan where a prefix of Tasks is already
  `done` skips them without invoking the engine; every one of the stop
  reasons above, each reached through a fixture Task/Review/Assignment setup
  that puts a Task in that exact repository state; that a second, immediate
  invocation against unchanged repository state reproduces the identical
  stop outcome without rerunning any completed Task or mutating anything
  beyond what the single engine invocation itself would mutate (idempotent
  re-invocation); and that a Task in an undecidable or failed state is never
  skipped in favor of a later Task in the order.
- Existing `LoopEngine`/`engine.test.js` behavior for direct single-Task
  `run` calls is unchanged except for the additive `category` field present
  on `"halted"` outcomes.

## Constraints

- Implement as an internal library module (for example alongside
  `src/engine.js`/`src/plans.js`), not wired into `src/cli.js` or
  `src/dashboard.js` in this proposal — that is later proposals' work.
- Do not introduce a new persisted execution-run document, lock file, or
  Task dependency field; derive all state from the adopted plan's `PLAN.md`
  and existing Task/Review documents.
- Do not duplicate any responsibility `LoopEngine` already owns (Role
  dispatch, Review projection, Approval handling, capacity continuation,
  retry limits, session recording); the only allowed engine change is the
  additive outcome-category extension described above.
- No background daemon, watcher, or concurrent scheduler; the runner
  advances at most one Task per invocation of the underlying engine call it
  makes, and returns control to its caller on any non-`done` outcome.

## Context

`src/engine.js`'s `LoopEngine.run(taskId, options)` already advances one
Task through Role dispatch, Review projection, Approval handling, capacity
continuation, and retry limits, returning outcomes shaped like `{kind:
"done" | "blocked" | "halted" | "waiting", task, reason?, retryAt?}`.
`src/plans.js`'s `inspectPlan`/`adoptPlan` already parse and validate
`PLAN.md`'s Execution Order section and its placeholder-to-real-id rewrite;
this proposal's ordering derivation should read the same adopted-form
section rather than re-implementing Markdown section parsing. `src/
documents.js`'s `TaskStore` already loads and validates individual Tasks
the same way the engine does. This proposal is the foundation every other
proposal in this plan consumes.

### Attempt 1

#### Summary

Added the additive `category` field to LoopEngine's `halted` outcomes in src/engine.js (one of `approval_gate`, `worker_failure`, `invalid_document`, `conflict`, `cancelled`, derived from role/error-type context, never from reason-string matching), extended cancellation detection to cover active Worker execution (not just capacity-wait sleep), exported `markdownSection` from src/plans.js for reuse, and created src/progression.js with `readPlanOrder` (derives ordered real Task ids from an adopted PLAN.md's Execution Order section, rejecting unadopted P-## placeholders and Task/plan project mismatches without partial results), `selectNextTask` (finds the first non-done Task, skipping a done prefix, reporting plan completion when all are done), and `runProgression` (the main entry point that repeatedly invokes `LoopEngine.run` once per Task, continuing on `done` and stopping on any other outcome with a mapped stop reason, completed list, and operator action string). ### Attempt 2

#### Summary

Fixed both acceptance-blocking defects from review-0021. (1) src/engine.js: removed the blanket rule that classified every halt during approver dispatch as approval_gate; haltedWorker() is now always worker_failure, and approval_gate is applied only at the single call site where the approver Role's own Result has status "failure" (missing/invalid decision file) — assignment-resolution failures, execution errors/timeouts, capacity failures, and malformed Results during approver dispatch now correctly halt as worker_failure. (2) src/progression.js: runProgression no longer rescans repository state via selectNextTask after each `done` outcome; it now calls selectNextTask once to establish the resume point (skipping an already-done prefix), then walks the plan order by index, appending to an in-call completed list and advancing the index only after an engine.run `done` outcome, guaranteeing each Task is passed to the engine at most once per call. Updated test/progression.test.js: replaced the no-approver-assignment fixture (which masked defect #1) with a faithful HumanApprover fixture that reuses workers/human-approver.mjs's real readDecision() against an actual missing decision file; added a test proving an approver Assignment-resolution failure and an approver execution failure both halt as worker_failure, not approval_gate; and strengthened the idempotent-reinvocation test with an already-done task-0001 prefix, asserting the approver Worker is invoked only for task-0002 on both runs (never re-invoked for the completed Task) and that neither Task file is mutated across two immediate invocations.

#### Verification

Ran `npm test` (node --test): all 169 tests pass (168 prior + 1 new), including the new approver-misclassification coverage and the strengthened idempotence test with a done prefix; no regressions elsewhere. Verified with `node --check` that src/engine.js, src/progression.js, test/engine.test.js, and test/progression.test.js are syntactically valid.
