# Plan cross-repository bootstrap and brief intake

## Objective

Produce a reviewed `software-feature` plan under `plans/cross-repo-operation/`
that lets an operator point AIOS at any other repository and start work with a
single natural-language objective. Today the engine already accepts `--root`,
Workers already run in that root, and the Planner convention already turns one
brief into an adopted plan; what is missing is the entry point. The plan must
close two gaps: bootstrapping a target repository's `.aios/` operating state
(`init`), and materializing an operator objective as a valid planning Task
without hand-writing the document every time (`brief`).

## Acceptance Criteria

- `PLAN.md` selects the `software-feature` profile and explains how the
  contracts, CLI surface, safety boundaries, and end-to-end verification shape
  the decomposition.
- The plan defines an `init` capability that scaffolds a target repository for
  AIOS operation: the `.aios/` layout the engine, dashboard, and progression
  already expect, plus machine-local worker assignment (or routing)
  configuration. It validates the target is a repository root, is idempotent,
  never overwrites existing operator state, and offers a no-write check mode
  consistent with `adopt --check`.
- The plan defines a `brief` capability that takes one operator objective and
  materializes exactly one planning Task in the target repository's
  `.aios/tasks/`: next sequential id, valid `aios.task/v1` front matter, the
  objective preserved verbatim as the brief, and instructions that scope the
  Implementer to writing `plans/<plan-id>/` in that same repository under the
  existing Planner protocol and profile table. It creates the Task document
  only; it never dispatches a Worker, spends provider capacity, or modifies
  existing Tasks.
- Planning Tasks generated by `brief` default to `approval: required`, and the
  generated wording must not grant the Planner authority beyond the existing
  convention: proposals still only become Tasks through operator-invoked
  adoption.
- Worker adapter references in bootstrapped configuration are explicit and
  portable: the plan states how a target repository locates the adapters
  (absolute operator-confirmed paths or an equivalent explicit mechanism), and
  committed documents in the target repository remain provider-neutral with no
  user-local paths or secrets, matching how `.aios/assignments.json` is treated
  here.
- The full loop is preserved unchanged: after `init` and `brief`, existing
  `run`, `adopt`, `progress`, and `dashboard` commands drive the target
  repository end to end via `--root` with no aios-lab-specific assumptions
  (project id, directory names, or hard-coded paths) leaking into behavior.
  Existing single-repository usage without `init` remains valid.
- Validation fails closed before any write: a target that is not a repository
  root, already-initialized state that conflicts, an empty or oversized
  objective, or an id collision each produce a distinct usage error and leave
  the target untouched.
- Verification includes a disposable end-to-end demonstration: bootstrap a
  scratch repository, submit a brief, run the generated planning Task through
  the real loop, adopt the resulting plan, and progress at least one adopted
  Task, showing documents, reviews, ledgers, and dashboard output land in the
  target repository rather than in aios-lab.
- The decomposition keeps contracts/validation, `init`, `brief`, and the
  end-to-end proof in focused Tasks where that improves one-session
  reviewability, with ordering in `PLAN.md` only.
- `node src/cli.js adopt plans/cross-repo-operation --check` passes without
  writing Tasks, and the Implementer's Verification records that command and
  its result.

## Constraints

- This is a planning Task. The Implementer writes only under
  `plans/cross-repo-operation/`; it does not modify `.aios/`, `src/`, `test/`,
  workers, assignments, or git history.
- Do not implement the commands in this Task, and do not propose a daemon,
  concurrent scheduling, provider SDKs, network services, or automatic
  dispatch of paid Worker sessions from `init` or `brief`.
- The plan must reuse the existing Planner convention, profiles, adoption, and
  progression semantics rather than defining a parallel planning pipeline.
- This Task carries `approval: required` because it defines how AIOS gains
  write access to repositories beyond its own.

## Context

The engine, progression, dashboard, and adoption commands all accept `--root`,
`CommandWorker` spawns Workers with the target root as their working
directory, and document validation checks project consistency rather than any
fixed project id, so the runtime can already operate on another repository.
In practice it cannot be used that way: the operator must hand-create
`.aios/assignments.json` and hand-write a planning Task document in the target
repository before anything runs. The Planner convention (task-0012) and
adaptive routing (tasks 0024-0026, merged) make the loop itself strong enough
that the entry point is now the limiting factor. Remaining routing tasks
0027-0029 are independent polish and neither block nor depend on this plan.
