# Prove cross-repository operation end to end and document it

## Objective

Demonstrate that the bootstrap and intake commands complete the
cross-repository story with the loop itself unchanged: a disposable scratch
repository goes from empty to a progressed adopted plan using only the
public CLI via `--root`, with every document, review, ledger, and dashboard
artifact landing in the scratch repository rather than in aios-lab. Add the
regression coverage that pins the unchanged single-repository behavior and
the operator documentation for the new workflow.

## Acceptance Criteria

- A disposable end-to-end demonstration, runnable as an automated test or
  scripted fixture, performs in a scratch git repository under a temporary
  directory: `init` with a configuration whose adapters are deterministic
  scripted command Workers satisfying the portability rules; `brief` with a
  natural-language objective; `run` (or `progress`) driving the generated
  planning Task through Implementer, Reviewer, and its required approval in
  the real Loop Engine; an operator-style adoption of the produced plan;
  and `progress` completing at least one adopted Task.
- The demonstration asserts placement: generated Tasks and Reviews under
  the scratch repository's `.aios/`, approval evidence under its
  `.aios/approvals/`, session ledger entries under its `.aios/runtime/`,
  the produced plan under its `plans/`, and dashboard output at its root
  when `dashboard --root` runs against it. It also asserts that aios-lab's
  own `.aios/` and working tree are byte-for-byte unchanged by the whole
  demonstration.
- The scratch repository uses a project id, directory name, and paths that
  differ from aios-lab's, proving no aios-lab-specific assumption leaks
  into behavior; the demonstration consumes no paid provider capacity and
  requires no network access.
- Regression coverage proves existing single-repository usage without the
  bootstrap step remains valid: the current test suite still passes
  unmodified in behavior, and a test exercises `run` and `dashboard`
  against a hand-prepared repository that never saw `init`.
- The README gains a cross-repository operation section documenting: the
  bootstrap command, its idempotence, check mode, and exit codes; how
  worker adapters are located through explicit operator-confirmed paths
  and why the machine-local configuration stays out of the target's
  version control; the intake command, its validation failures, and the
  approval default on generated planning Tasks; the unchanged
  `run`/`adopt`/`progress`/`dashboard` flow via `--root`; and the
  authority boundary — Workers propose under `plans/`, only the engine
  writes `.aios/`, and adoption is operator-invoked.
- Usage output in the CLI lists both new subcommands with their flags and
  exit codes alongside the existing commands.
- `npm test` passes for the whole repository, including the new
  demonstration, and the demonstration cleans up its temporary directory.

## Constraints

- No engine, progression, adoption, dashboard, or contract behavior
  changes: this Task delivers demonstration fixtures, tests, and
  documentation, plus at most cosmetic CLI usage text.
- The demonstration must be disposable and deterministic: scripted command
  Workers only, no paid provider sessions, no network, no daemon, and no
  background processes left running.
- Do not write into aios-lab's `.aios/` from any Worker or fixture; all
  generated operating state belongs to the scratch repository.
- No new runtime dependencies; test-time helpers follow the existing
  fixture patterns.

## Context

`fixtures/progression-demo.js` and the existing progression and adoption
end-to-end tests show the established pattern for driving the real engine
with scripted command Workers in a disposable directory. The README already
documents `run`, capacity waiting, `progress`, the dashboard, the Planner
convention, and adaptive routing; the new section completes that operator
manual. This Task is last because it verifies the assembled feature against
the contracts the earlier work stabilized and writes documentation that
matches the implemented surface.
