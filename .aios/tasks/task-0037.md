# Define a validated candidate-cooldown contract and selection gate

## Objective

Introduce a strict, machine-local per-candidate cooldown record (candidate id,
`retry_at`, bounded sanitized evidence) and extend the pure adaptive-routing
selection engine to treat a cooled-down candidate as ineligible with an
explicit, auditable gate reason — entirely through validated function inputs,
with no file I/O, clock read, or other side effect added inside selection
itself.

This Task carries `approval: required` because it adds a new non-overridable
hard safety gate to the adaptive-routing selection engine and its matching
ledger validation.

## Acceptance Criteria

- A new strict schema (for example `aios.candidate-cooldowns/v1`) defines a
  cooldown record: a candidate id matching the existing catalog candidate-id
  shape, an ISO `retry_at` timestamp, and a bounded, sanitized evidence string
  produced with the same redaction rules `src/routing-ledger.js` already
  applies to failure diagnostics (secrets and local paths redacted, bounded
  length, no control characters), tied to the failure reason code that
  justified the cooldown. Validation exact-keys every record and the
  top-level document the same way existing routing schemas do, rejecting any
  unrecognized field.
- `selectCandidate` in `src/routing-policy.js` gains a new optional input
  (for example `cooldowns`) validated the same way `history` and `recovery`
  already are. Selection never reads a clock, file, or environment variable
  itself; the caller supplies both the cooldown records and the "as of"
  observation time already resolved.
- A new hard-gate reason code (for example `candidate_cooldown_active`) is
  added to `SELECTION_REASON_CODES` in a defined position within the existing
  fixed gate order. A candidate whose supplied cooldown `retry_at` is strictly
  after the supplied observation time is marked ineligible with that reason;
  a candidate whose `retry_at` is at or before the observation time is treated
  as not cooled down, so expiry is automatic from the caller's point of view
  and requires no separate cleanup step to stop affecting selection.
- `src/routing-ledger.js`'s considered-candidate validation (its `GATE_CODES`
  set and the fixed-order check in `validateConsidered`) is updated in the
  same change with the new reason code, so a routing-decision record naming a
  cooldown skip validates instead of failing ledger load or record, avoiding
  the exact class of symmetric-validation gap fixed in commit `11a84e0`.
- The considered-candidate row for a cooled-down candidate keeps the existing
  shape (`candidate`, `provider`, `model`, `tier`, `eligible`, `reasons`), so
  existing dashboard and ledger consumers require no change to render it.
- Cooldown gating composes correctly with every existing gate and with
  distribution: a cooled-down candidate is excluded before fitness/
  distribution exactly like `candidate_disabled`, and — because the new reason
  code is not added to `OVERRIDE_DISPLACEABLE_REASON_CODES` — an operator
  override cannot restore a cooled-down candidate the way it can restore a
  cost- or latency-rejected one.
- Unit tests over fakes and fixture configurations prove: a candidate with a
  future `retry_at` is ineligible with exactly the new reason code and no
  other; a candidate whose `retry_at` has passed is fully eligible; a cooldown
  on an otherwise-winning candidate correctly reroutes selection to the next
  best eligible candidate; and a routing-decision record whose considered row
  carries the new reason code passes `validateDecisionRecord`.

## Constraints

- Do not add file I/O, a clock read, or any other side effect inside
  `src/routing-policy.js`; cooldown data and the observation time must arrive
  as validated function inputs exactly like `history` and `recovery` do today.
- Do not change the meaning or fixed position of any existing
  `SELECTION_REASON_CODES` entry, and do not add the new reason code to
  `OVERRIDE_DISPLACEABLE_REASON_CODES`.
- Do not implement cooldown persistence, automatic creation from capacity
  failures, expiry pruning of a stored file, or any CLI command in this Task;
  those belong to a separate Task in this plan.
- Do not change the `aios.routing/v1` candidate catalog schema; a cooldown is
  separate machine-local runtime state, not routing configuration.

## Context

`selectCandidate` (`src/routing-policy.js`) is a deliberately pure function
operating only on validated inputs, and `src/routing-ledger.js` mirrors its
gate vocabulary (`GATE_CODES` sourced from `SELECTION_REASON_CODES`) so every
considered row's reasons validate consistently. Dogfooding finding 3 observed
that after a Codex capacity exhaustion, every subsequent action could
reselect the same exhausted candidate, and the only available lever was
hand-editing the catalog's `enabled` flag — which itself caused the
policy-revision conflict recorded as finding 4. This Task builds only the
contract and the pure gate; wiring it into real dispatch and exposing it to
the operator is a separate, later Task in this plan.
