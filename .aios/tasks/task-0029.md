# Prove adaptive routing end to end and document operations

## Objective

Prove the assembled adaptive routing feature with durable integration tests and
a reproducible disposable demonstration that use fake Claude and Codex command
Workers, never paid sessions. Exercise high-tier planning, lower-tier bounded
implementation, equal-or-higher cross-provider review, weighted distribution,
operator override, capacity/provider fallback, failure escalation, sequential
plan progression, audit/session correlation, and dashboard rendering; then
document safe configuration, operation, recovery, and legacy compatibility.

## Acceptance Criteria

- A committed disposable demo harness creates a temporary AIOS root, routing
  configuration, fake command Workers, planning/general Tasks, Reviews, an
  adopted multi-Task plan, and clean routing/session ledgers. It invokes the real
  CLI entry points as child processes with exact argv arrays and removes the
  temporary root in `finally`, reporting setup, commands, observations, and
  cleanup without machine-local paths.
- Fake candidates declare explicit Claude and Codex provider ids and distinct
  lower/high model ids but perform no network or paid call. Their deterministic
  modes can complete, request changes, report typed verification/context
  failure, defer capacity with a candidate-bound continuation, time out, and
  report provider failure while emitting contract-valid session telemetry.
- One scenario proves a planning Task can select only a high-tier candidate even
  when a cheaper/lower candidate and provider distribution deficit favor the
  lower option. The audit row records the lower candidate's hard-gate rejection.
- One scenario proves a well-decomposed low-risk implementation selects an
  eligible lower-tier Implementer, then an equal-or-higher Reviewer from the
  other provider. A companion fixture changes one workload signal to unknown or
  high risk and proves the choice rises to high tier.
- Distribution fixtures run enough independent equivalent decision keys to
  demonstrate convergence to configured Claude/Codex weights with deterministic
  tie behavior. Re-resolving an old key after the window changes proves its
  candidate does not drift.
- Recovery scenarios prove: Claude capacity or provider failure falls back to
  an equivalent Codex candidate before promotion; a typed verification/context
  failure promotes to a strictly stronger unused candidate; a rejected Review
  promotes the next existing Implementer attempt without resetting retry count;
  duplicate evidence cannot loop; a foreign continuation is never sent across
  providers; and configured route limits end in the documented stop outcome.
- Override scenarios prove exact/wildcard and configured/CLI precedence, audited
  displacement of the normal winner, hard-gate refusal of an unsafe Planner or
  Reviewer downgrade, and fallback-allowed versus fallback-denied behavior.
- An adopted two-Task plan completes through exactly one real `aios progress`
  invocation using routing. The test asserts Task order, Role decisions, Review
  provider separation, final Task states, routing/session ledger correlation,
  CLI routing summaries, and dashboard evidence generated from the same root.
- A legacy fixture runs the same `aios.assignments/v1` command path before and
  after the feature and asserts unchanged stdout, exit codes, Task transitions,
  session records, capacity behavior, and absence of a routing ledger.
- README documentation covers: routing versus legacy Assignment mode; the full
  candidate/policy/hint/override schema with portable placeholder commands;
  capability tiers and the Planner high-tier invariant; lower-tier eligibility;
  Reviewer tier/provider rules; distribution's fitness-first boundary; decision
  keys and audit location; CLI overrides; fallback/escalation order and limits;
  continuation safety; dashboard interpretation; capacity recovery; and how to
  disable routing by returning to legacy assignments.
- Documentation explicitly states that model availability, pricing, and names
  are operator-maintained configuration rather than source-code truth; examples
  do not promise that a named commercial model is currently available and do
  not contain credentials or user-local paths.
- The demo command, focused integration tests, and full `npm test` pass. The
  Verification records exact commands, scenario outcomes, test totals, and
  confirmation that no network/paid provider process was launched and the
  temporary root was removed.

## Constraints

- Use fake command Workers only. Do not invoke installed Claude/Codex CLIs,
  network APIs, credentials, or paid sessions in tests or the demo.
- Keep fixtures deterministic and platform-portable; launch Node with
  `process.execPath`, pass argv without a shell, use OS temporary directories,
  and clean up in `finally`.
- Do not add new production routing behavior merely to satisfy the demo; defects
  found belong in the owning production module and require focused regression
  coverage.
- Do not introduce Task dependencies, concurrent scheduling, a daemon, remote
  capacity polling, or third-party dependencies.

## Context

AIOS already has durable command-worker fixtures, disposable plan-progression
demonstrations, strict Task/Review/Plan contracts, session telemetry, and an
offline dashboard. A single mixed-provider simulation should assemble those
real surfaces so routing quality, workflow correctness, auditability, and legacy
compatibility are demonstrated together without relying on live provider
availability.
