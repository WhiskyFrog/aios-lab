# Show adaptive routing evidence on the dashboard

## Objective

Extend the existing self-contained offline dashboard with a read-only Routing
Decisions view. An operator can see which provider/model/tier was chosen for a
Task and Role, why alternatives were rejected, whether distribution, override,
fallback, or escalation affected the route, and how actual provider share
compares with configured targets, while keeping routing decisions distinct from
session usage/capacity telemetry and from the Task's current lifecycle state.

## Acceptance Criteria

- Dashboard data loading uses the routing ledger's sanitized projection API; it
  does not re-parse the raw JSON schema or reproduce selection math in rendering
  code. A missing routing ledger produces an explicit empty state. An invalid
  ledger produces a visible error naming the path and validation reason without
  preventing Tasks, Plans, Reviews, or Worker Sessions from rendering.
- A Routing Decisions summary shows counts by provider, model, tier, and Role for
  the configured finite distribution window, configured target shares, actual
  shares, and signed deficit/surplus. It labels the data as historical decisions,
  not live provider capacity or guaranteed future distribution.
- Each Task with routed history shows its latest decision per active Role and
  expandable ordered steps. The compact view includes candidate/provider/model/
  tier, policy revision, decision time, outcome, and badges for planning-high,
  lower-tier-eligible, cross-provider review, same-provider exception, override,
  distribution-changed-winner, fallback, escalation, and exhausted route when
  applicable.
- Expanded evidence lists normalized workload bands and their sources; every
  considered candidate with eligible/ineligible reason codes; the fitness tuple;
  provider target/window/deficit used; normal winner versus override winner;
  parent/step relation; and sanitized fallback/escalation reason. It never shows
  argv, executable paths, environment, credentials, prompt/Task bodies,
  continuation tokens, or unsanitized provider stderr.
- Routing and session evidence remain visibly separate and linked only by the
  sanitized session id when one exists. Routing answers "why this candidate";
  Worker Sessions remains the authority for usage, cost, outcome, capacity, and
  refill. The dashboard does not sum usage from both or infer capacity for a
  candidate with no session.
- Task current state and plan progression remain derived from Task/Plan documents
  exactly as today. A historical failed or exhausted route is never presented as
  proof that the Task is currently failed, waiting, or blocked after a later run.
- Reviewer rows visibly identify the compared Implementer provider/tier and
  whether different-provider preference was achieved. A same-provider exception
  shows the recorded reason that every eligible alternative failed a higher
  priority gate.
- Planning decisions below the configured high tier or Reviewer decisions below
  their Implementer tier are rendered as integrity-error cards rather than
  normal badges, even if malformed data somehow reaches the projection. The
  dashboard remains read-only and does not repair the ledger.
- The view works at narrow mobile widths without horizontal page scrolling;
  dense candidate evidence uses accessible disclosure/table patterns, visible
  focus, semantic headings, non-color-only badges, and readable long identifiers.
  Existing dashboard sections and navigation remain intact.
- Tests cover missing/invalid ledgers, provider-share calculations from the
  projection, each badge/evidence state, sensitive-field absence, routing/session
  separation, integrity errors, stale-history labeling, HTML escaping,
  accessibility structure, responsive CSS, and unchanged output when no routing
  data exists beyond the documented empty section.

## Constraints

- Keep `aios dashboard` a one-shot read-only generator. Do not add start/stop,
  retry, override, edit, or repair controls; JavaScript is not required for core
  content.
- Do not reimplement policy evaluation, distribution selection, ledger
  validation, Task progression, or session accounting in dashboard code.
- Do not expose command paths, environment variables, credentials, raw prompts,
  continuation tokens, or provider diagnostics.
- Do not add a server, watcher, daemon, concurrency, remote provider calls, or a
  front-end dependency.

## Context

The current dashboard already renders Tasks, Reviews, plan progress, next
actions, and Worker Sessions from strict repository documents. Routing decisions
are a new historical evidence source with a different purpose from session
telemetry. Preserving that distinction lets an operator audit model choice
without mistaking a past dispatch for live Task or provider state.
