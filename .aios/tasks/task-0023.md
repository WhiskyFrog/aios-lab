# Plan adaptive model routing across Claude and Codex

## Objective

Produce a reviewed `software-feature` plan under
`plans/adaptive-model-routing/` for auditable Task-and-Role-aware model
routing. The policy must reserve strong models for planning and difficult or
high-risk work while allowing well-decomposed, low-risk implementation Tasks
to use appropriately lower-cost models without weakening review or recovery.

## Acceptance Criteria

- `PLAN.md` selects the `software-feature` profile and explains how routing
  contracts, policy evaluation, compatibility, operational evidence, and
  end-to-end verification shape the decomposition.
- Planner work has an explicit high-capability invariant. A low-tier Planner
  must not be selected merely to save cost; lower tiers become eligible only
  for sufficiently bounded downstream Roles after a strong Planner has made
  the work small, clear, and verifiable.
- The plan defines a configured provider/model candidate catalog with
  capability tiers and declared availability. The router does not silently
  invent model identifiers or infer capabilities from names.
- Selection evaluates Task and Role evidence including plan profile,
  complexity, risk, context size, tool needs, verification burden, latency,
  cost budget, and prior outcomes. The policy is deterministic for the same
  repository state and configuration.
- The current Role-only `aios.assignments/v1` behavior evolves through a
  backward-compatible Task-and-Role-aware resolver. Existing assignments,
  single-Task runs, plan progression, session records, and approval behavior
  remain valid.
- Implementer and Reviewer use different providers when eligible, and the
  Reviewer is assigned an equal-or-higher capability tier for the work being
  checked. Materially equivalent candidates follow a configurable
  distribution target so work does not silently concentrate on Claude or
  Codex.
- Escalation and fallback are bounded, loop-safe, and testable. Capacity,
  timeout, or provider failure tries a declared equivalent cross-provider
  fallback before a higher tier where policy permits; test failure, review
  rejection, repeated evidence, or repeated context failure can promote the
  next attempt to a stronger model.
- Every routing decision records the chosen provider/model and tier,
  considered alternatives, selection rationale, fallback or escalation
  rationale, and distribution-policy effect in an operationally appropriate
  ledger or a newly justified durable record. Provider choices do not leak
  into adopted Task documents.
- Operators retain explicit per-run or configured overrides, with validation
  and audit evidence showing when an override displaced the policy choice.
- The decomposition keeps candidate configuration, policy resolution,
  engine/CLI integration, observability, and verification in focused Tasks
  where that improves one-session reviewability. It does not introduce
  concurrent scheduling.
- Verification covers deterministic selection, Planner tier enforcement,
  lower-tier eligibility, cross-provider review, distribution behavior,
  override handling, escalation and fallback termination, backward
  compatibility, and a disposable end-to-end routing demonstration.
- `node src/cli.js adopt plans/adaptive-model-routing --check` passes without
  writing Tasks, and the Implementer's Verification records that command and
  the focused plan-validation result.

## Constraints

- This is a planning Task. The Implementer writes only under
  `plans/adaptive-model-routing/`; it does not modify `.aios/`, `src/`,
  `test/`, workers, assignments, or git history.
- Do not implement a concurrent scheduler, daemon, provider SDK, or routing
  behavior in this Task. The plan may integrate with automatic sequential
  plan progression but must preserve its existing stop and approval gates.
- Keep adopted Task documents provider-neutral. Provider/model inventory,
  credentials, capacity, distribution targets, and overrides belong in
  operator configuration or runtime policy, never hard-coded user-local paths
  or secrets.
- Prefer explicit, inspectable rules over an opaque model-scored router. Any
  learned or heuristic scoring proposed later must retain deterministic guard
  rails and explainable decision evidence.
- This Task carries `approval: required` because the plan changes how AIOS
  chooses execution resources and spends provider capacity.

## Context

Sequential inter-Task progression and its dashboard/CLI visibility are now
implemented and verified independently. The current assignment mechanism is
still Role-only, and local Planner experiments pair a strong Claude
Implementer with a strong Codex Reviewer. Fable may be used opportunistically
when capacity is available, with Sonnet as the same-provider fallback, but
those local wrappers are temporary evidence rather than the future routing
contract. The next milestone is to turn that operating practice into a
provider-balanced, auditable engine policy while preserving strong planning
and review.
