# Show adopted plan progression status on the dashboard

## Objective

Extend the existing self-contained, offline `aios dashboard` with a
read-only "Plan Progress" section that reports, for every fully adopted
plan, its ordered Tasks' completion status and current Task, re-derived
fresh from the same Task/plan documents the core progression library and
`aios progress` already read — never from any new persisted record. The
section keeps two kinds of information visibly distinct and separately
labelled: **current state**, computed live from the repository on every
dashboard run, and **last-observed session evidence**, drawn from the
existing session ledger and explicitly labelled as historical rather than
live. It never implies it can reconstruct a transient stop event — one that
left no mark in either the Task document or the session ledger — after the
process that produced it has ended. The dashboard remains a one-shot,
read-only pass with no button, link, or other control that starts, stops, or
otherwise mutates progression.

## Acceptance Criteria

- For every plan directory under `plans/` that is fully adopted (no `P-##`
  placeholders remain, consistent with `collectPlanProposals`'s existing
  adopted/pending distinction), the dashboard shows its ordered real Task
  ids, how many are `done`, and the current (first non-`done`) Task, using
  only the core progression library's ordering/derivation — it does not
  re-parse `PLAN.md` or re-implement Task-order extraction.
- When every Task in a plan's order is `done`, the section shows that plan
  as complete.
- When the plan is not complete, the section reports the current Task's
  state as one of exactly three **current-state** categories, each
  re-derived live from that Task's own document and never from history:
  awaiting approval (Task `state: approval`), blocked (Task `state:
  blocked`, further labelled rejected or retry-limit-exhausted from the
  Task's own `approval` and `retry` fields), or invalid document (the Task
  or a Review referenced by the plan's order currently fails to load or
  validate). For each of these three, the section also shows the same
  operator action string `aios progress` would report for that exact Task
  state, computed by the same core-library derivation, so an operator
  reading only the dashboard knows what to do without separately running
  `aios progress`.
- When the current Task is not in one of those three current-state
  categories (for example, it is mid-`implement` or mid-`review` with no
  distinguishing marker in the Task document itself), the section may
  additionally show, in a visibly separate area labelled "last observed",
  the most recent session-ledger row for that Task (keyed by Task id and
  Role, reusing `src/sessions.js`'s existing ledger) if one exists: its
  outcome (`failed` or `capacity_deferred`) and timestamp. This is
  historical evidence, not current status — it may be stale, since a newer
  attempt not yet reflected in the ledger could already have changed the
  Task's state — and the section never attaches an operator-action string
  to it, because acting on a possibly-stale historical event without the
  CLI's live confirmation would risk misleading the operator. When the
  current Task is not in a current-state category and no session-ledger row
  exists for it either, the section shows neither a current-state nor a
  last-observed classification for that Task.
- The section never reports cancellation or a repository-mutation conflict,
  under any label, current or historical. Neither is recorded in the Task
  document, and the session ledger's outcome vocabulary is only
  `completed`/`failed`/`capacity_deferred` — moreover, the `approver` Role's
  own Worker executions are never written to the ledger at all — so once the
  process that produced either event has ended, the dashboard has no
  evidence for it. Both remain observable only in `aios progress`'s own live
  output at the moment they occur; the dashboard must not imply otherwise.
- A plan directory or Task document that fails to load or fails core-library
  validation is rendered as a visible, named error card naming the plan and
  the reason, consistent with the existing dashboard's error handling, and
  does not abort collection of the other plans or the rest of the
  dashboard.
- The section adds no interactive control: no form, button, link, or script
  that starts, stops, retries, or otherwise mutates a Task, a plan, or any
  progression state; the dashboard stays a static, offline HTML file
  generated by one command with no server, daemon, or watcher.
- Layout and accessibility are consistent with the existing dashboard:
  correct heading hierarchy, landmark region placement, and CSS-only
  responsive collapsing at narrow widths; status is not conveyed by color
  alone.
- Automated tests cover: a complete adopted plan; an in-progress adopted
  plan whose current Task is in each of the three current-state categories
  (awaiting approval, blocked-rejected, blocked-retry-limit-exhausted), each
  asserting the matching operator action string renders; a current Task
  with no current-state marker but a matching session-ledger row present,
  asserting the last-observed area renders with its historical label,
  outcome, and timestamp, and no operator-action string; a current Task with
  neither a current-state marker nor a session-ledger row, asserting neither
  classification renders; a plan with no Tasks yet run; an adopted plan
  whose Task order fails to load; and, across every fixture, that the
  rendered section never mentions cancellation or a repository conflict
  under any label and contains no interactive/mutating markup.

## Constraints

- Read-only: this proposal does not adopt, create, modify, or delete
  anything under `plans/`, `.aios/`, or elsewhere in the repository, and it
  must not run `LoopEngine.run` or otherwise invoke a Worker — it only reads
  Task/Review/plan documents through the same read paths the core library
  and existing dashboard already use.
- Reuse the core progression library's ordering, resume, and stop
  classification; do not duplicate that logic in `src/dashboard.js` or
  `src/plan-dashboard.js`.
- Do not label last-observed session-ledger evidence as current state, and
  do not display cancellation or repository-mutation-conflict information
  under any label — neither is ever recorded in a document or the session
  ledger, so there is no honest way to report either after the fact.
- Add no new runtime dependency; no required JavaScript for the core view.

## Context

`src/dashboard.js` already renders Task lifecycle cards and, via
`src/plan-dashboard.js`'s `collectPlanProposals`, a "Plan proposals awaiting
adoption" section listing not-yet-adopted plans. This proposal adds the
complementary view for already-adopted plans' in-flight ordered progress,
consuming the core progression library delivered earlier in this plan
instead of re-deriving plan order or stop status. The README documents the
dashboard as strictly read-only and one-shot; this proposal must remain
consistent with that description, updating the README's Dashboard section
to describe the new "Plan Progress" section.
