# Provide a Codex worker adapter

## Objective

Prove the "Roles are permanent, workers are replaceable" design across
vendors: a command Worker adapter that dispatches any Role to one
non-interactive OpenAI Codex session, interchangeable with the Claude
adapter through Assignment configuration alone.

## Acceptance Criteria

- `workers/codex-worker.mjs` implements the command Worker contract: Task
  document on stdin, `AIOS_TASK_ID` and `AIOS_ROLE` in the environment,
  exactly one `aios.result/v1` JSON object on stdout for a usable session
  reply.
- Role prompts and payload validation are semantically identical to the
  Claude adapter's. Logic shared by both adapters (role prompts, payload
  extraction, payload validation, failure discipline) is factored into a
  common module under `workers/` instead of being duplicated; the Claude
  adapter's observable behavior is unchanged and all existing tests keep
  passing after the refactor.
- Each Role action launches exactly one `codex exec` session with a
  role-appropriate sandbox: `--sandbox workspace-write` for the
  Implementer, `--sandbox read-only` for Reviewer and Approver. The
  dangerous bypass flags (`--dangerously-*`) are never used.
- The reply is read deterministically from a temporary
  `--output-last-message` file (cleaned up afterwards) and interpreted by
  the shared extraction rule: a valid role payload becomes a
  `status: success` Result, a `failure_reason` reply becomes
  `status: failure`, and anything unusable — including a nonzero
  `codex exec` exit — makes the adapter exit nonzero so the engine halts
  without a Task transition.
- The Codex launch command comes from the adapter's trailing argv tokens
  (one or more, so both a native binary path and a `node <cli.js>` pair
  work) or `AIOS_CODEX_CLI`; `AIOS_CODEX_MODEL`, when set, is passed as
  `--model`, otherwise the CLI's configured default model is used.
- Codex-specific behavior stays in `workers/`; nothing under `src/`
  changes.
- Unit tests under `test/` cover argument construction (sandbox per Role,
  model handling, launch-command resolution) and reply handling through
  the shared logic without invoking Codex or any network; `npm test`
  passes for the whole repository.
- An end-to-end run in a disposable sandbox repository reaches `done`
  through Codex-backed Implementer and Reviewer sessions from one
  `aios run` invocation.
- README gains a "Codex Workers" section: Assignment example, the Windows
  shim caveat, sandbox mapping, and a mixed-vendor example (different
  Roles bound to different providers).

## Constraints

- Do not modify anything under `src/` or the v1 document contracts.
- Do not modify `.aios/` documents, including `.aios/assignments.json`;
  binding real Roles to Codex is an operator decision after this Task.
- No new runtime dependencies.
- Capacity deferrals (`aios.worker-execution/v1`) are out of scope for
  Codex in this Task: the adapter returns a bare Result v1, which the
  command transport accepts unchanged.
- Do not commit or push from a Worker session.

## Context

`codex-cli 0.142.3` is installed. The PATH entry `codex` resolves to a
PowerShell shim, so Workers spawned without a shell must name the real
launcher: the native binary at
`C:\DogFoot\JS\NODE\node_modules\node_modules\@openai\codex\node_modules\@openai\codex-win32-x64\vendor\x86_64-pc-windows-msvc\bin\codex.exe`
(verified runnable) or the portable pair
`node C:\DogFoot\JS\NODE\node_modules\node_modules\@openai\codex\bin\codex.js`.
Relevant `codex exec` surface: prompt as argument or stdin,
`--sandbox read-only|workspace-write|danger-full-access`,
`-o/--output-last-message <file>` for the final agent message,
`--json` for JSONL events, `-m/--model`, `-C/--cd`,
`--skip-git-repo-check`. The Claude adapter
(`workers/claude-worker.mjs`) already exports `rolePrompt`,
`extractPayload`, and `validatePayload`; `test/claude-worker.test.js`
shows the test style, and `workers/human-approver.mjs` shows the
entry-point guard pattern. The engine treats any argv command uniformly
(`.aios/results/README.md`, `README.md` Command Worker section).

### Attempt 1

#### Summary

Added workers/worker-shared.mjs holding the logic common to both vendor adapters (role prompts, reply extraction, payload validation, WorkerFailure/fail failure discipline, readStdin), and refactored workers/claude-worker.mjs to consume it instead of duplicating that code (its exports and behavior are unchanged; it re-exports the shared functions the existing test suite imports). Added workers/codex-worker.mjs: a command-Worker adapter that runs one non-interactive `codex exec` session per Role, launches with `--sandbox workspace-write` for implementer and `--sandbox read-only` for reviewer/approver (never a `--dangerously-*` flag), resolves the launcher from trailing argv tokens or AIOS_CODEX_CLI, passes AIOS_CODEX_MODEL as `--model` only when set, reads the reply deterministically from a temp `--output-last-message` file that is always cleaned up, and emits a bare aios.result/v1 on stdout (success/failure/nonzero-exit exactly per the shared extraction/validation rule), exiting nonzero with no stdout for anything unusable. Added fixtures/codex-cli.js (a fake `codex exec` for tests) and test/codex-worker.test.js covering sandbox-per-Role mapping, launch-command resolution, exec argument construction (including a no-dangerously-flags check), reply handling via buildResult, and full-adapter subprocess runs. Added a 'Codex Workers' README section with an Assignment example, the Windows shim caveat, sandbox mapping, and a mixed-vendor example. No changes under src/ or .aios/.

#### Verification

npm test: 113/113 pass (98 pre-existing tests still green after the claude-worker refactor, plus 15 new codex-worker tests) — confirms argument construction, launch-command resolution, and reply handling in isolation from any real Codex/network call. Cross-checked every flag the adapter emits against the real installed `codex exec --help` (codex-cli 0.142.3). Built a disposable git sandbox repo with a Task and an Assignment binding both implementer and reviewer to codex-worker.mjs against the real codex.exe, and ran `aios run` end to end: verified the adapter correctly launches a real `codex exec` session (correct workdir, sandbox: workspace-write) and correctly halts the run with no Task transition on a nonzero `codex exec` exit. Not run to completion: the full 'reaches done through Codex-backed Implementer and Reviewer sessions' live run could not be completed in this session — this machine's Codex account hit a usage-limit lockout ('You've hit your usage limit... try again at 4:05 AM') reproducible across every model the CLI accepts for this account (gpt-5.5 works but is quota-blocked; gpt-5, gpt-5-codex, o3, gpt-4.1 are rejected outright as unsupported for this ChatGPT-account login), confirmed via direct `codex exec` calls outside the adapter, so it is an account-capacity condition rather than an adapter defect. Once quota resets, rerun: `node src/cli.js run <task-id> --root <sandbox> --assignments <sandbox>/.aios/assignments.json` with AIOS_CODEX_MODEL=gpt-5.5 in the Assignment's environment (or wait for the account's default model to be upgraded) to confirm the done transition.
