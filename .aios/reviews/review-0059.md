# Review of task-0041, Attempt 2

## Findings

The shared distributionHistoryDecisions rule in src/routing-ledger.js (~line 1264-1310) still diverges between selection-time (historyProjection, atPresentPosition=true, closure always visible) and validateLedger replay (atPresentPosition=false, closure visibility for a generation with no later same-key generation falls back to the tiebreak `boundaryRecord.recorded_at > supersededAt`, which is false on a tie). This means: if resetAction supersedes a generation for key A, and a subsequent selection is then made and recorded for a DIFFERENT key B whose recorded_at equals the reset's observed_at (a plausible real scenario when an automation batch stamps multiple operations with the same 'now' timestamp), historyProjection at selection time correctly excludes A's closed generation from B's history (giving distribution.observed=0 in my repro), but validateLedger's replay of B's own row does NOT see the closure as 'visible' (tie is not '>') and recounts A's row into the window — causing `ledger.record()` to throw the exact original error, 'Routing decision 1.distribution.observed: must equal the actual preceding ledger window', inside the very same commit call that just wrote the record. I reproduced this directly against the current working tree (repro script, since removed): seed+fail+reset key task-0099:implementer:1 at RESET_AT, then select+record a fresh decision for key task-0099:implementer:98 with recorded_at=RESET_AT (tied with the reset's observed_at) — `ledger.record()` throws immediately. This is the same failure class the Task exists to eliminate, just relocated to a cross-key tie the Attempt's four regression tests (same-key reset/reselection, cross-key-record-then-reset tie, ordinary cross-key, supersession-without-reset) do not cover — none of the added tests exercise reset-then-cross-key-record with a tied or non-strictly-increasing timestamp. This violates Acceptance Criterion 1's requirement that 'a record's self-reported distribution numbers are always exactly reproducible by replaying the ledger' and Acceptance Criterion 3's requirement that the rule exclude exactly the closed-generation rows 'as of each record's position in append order' — the current implementation substitutes a wall-clock strict-inequality heuristic for true append order when a boundary record doesn't itself extend the reset key's generation, and that heuristic gives the wrong (inconsistent-with-selection) answer on ties. Please close this gap — e.g. by having resetAction leave a definitive, order-preserving signal (or by using something already anchored to true commit/append order rather than observed_at/recorded_at comparison) so cross-key closure visibility at replay always matches what was visible to the actual selection call, including when timestamps tie or are otherwise not strictly ordered — and add a regression test for the reset-before-cross-key-record tied-timestamp case (the mirror of the existing 'distribution replay keeps a generation that was still open at an earlier append position' test, but with the reset happening first).
